# spanIQ V2 — Build Plan

> **One-liner:** Production monitoring for LLM outputs — continuous, deterministic, $0/trace. Detects prompt drift, model swaps, and output degradation in real-time using the same statistical primitives from V1.

**Prereq:** V1 is complete — 4 metrics, 71 tests, evaluate() pipeline, pytest integration. V2 builds on top of V1. No rewrites.

---

## What V2 Solves That V1 Cannot

V1 answers: "did my LLM pass this test?" — offline, batch, before deployment.

V2 answers: "is my LLM still behaving like it was when it passed?" — continuous, live, after deployment.

The gap V1 exposed (from the honest limitations section):

```
1. Baseline collection is manual — no tooling to build baselines from production
2. No production monitoring path — V1 is entirely batch/offline
3. No drift timeline — no way to answer "when did this start degrading?"
```

V2 closes all three.

---

## What V2 Ships

1. **Baseline Store** — collect, persist, version, and manage baseline output sets
2. **Trace Collector** — ingest LLM outputs from multiple sources (langfuse, JSONL files, direct SDK)
3. **Monitor Loop** — continuous evaluation against rolling baseline windows
4. **Drift Timeline** — SQLite-backed time series of metric scores with alerting
5. **3 Replay Demos** — prompt injection, model swap, RAG breakage — all reproducible with free LLM APIs
6. **CLI extensions** — `spaniq baseline collect`, `spaniq monitor run`, `spaniq timeline show`

## What V2 Does NOT Ship

```
❌ Dashboard / web UI (matplotlib PNG exports are the visual layer)
❌ Cloud platform / hosted service
❌ Agent-specific or RAG-specific metrics (still uses V1's 4 metrics)
❌ Multi-tenancy
❌ Auth / API keys
❌ Webhook alerting (stdout + JSONL log only)
```

## Architecture

### V1 → V2 Relationship

```
V1 (untouched):
  LLMTestCase → metrics → evaluate() → EvalResult

V2 (new layer on top):
  Trace Source → Collector → LLMTestCase construction → Monitor → metrics → TimelineStore → Alert
       ↑                          ↑
  langfuse API              BaselineStore
  JSONL file                (provides baseline_outputs
  direct SDK call            for each test case)
```

V2 does NOT modify any V1 code. It imports `evaluate()`, all 4 metrics, and `LLMTestCase` from the existing `spaniq` package and builds orchestration around them.

### Data Flow

```
1. COLLECT BASELINES
   User runs: spaniq baseline collect --source groq --prompt "refund policy?" --n 50
     → calls Groq API 50 times with the same prompt
     → stores 50 outputs in BaselineStore (SQLite)
     → tagged with: prompt_hash, model_name, collected_at, version

2. INGEST TRACES
   Source A: spaniq monitor run --source langfuse --project my-chatbot
     → polls langfuse Observations API v2 every 30s
     → extracts (input, output) pairs from GENERATION spans

   Source B: spaniq monitor run --source file --path traces.jsonl
     → reads JSONL file line by line (tail -f mode)
     → each line: {"input": "...", "output": "...", "timestamp": "..."}

   Source C: Python SDK
     from spaniq.monitor import Monitor
     mon = Monitor(baseline_id="refund-v1")
     mon.ingest(input="refund policy?", output="we offer 30-day refunds")

3. SCORE EACH TRACE
   For each ingested trace:
     → look up baseline_outputs by prompt_hash from BaselineStore
     → construct LLMTestCase(input, actual_output, baseline_outputs)
     → call evaluate([tc], [configured_metrics])
     → get scores

4. STORE IN TIMELINE
   → write (trace_id, timestamp, metric_name, score, passed) to SQLite
   → indexed on (timestamp, metric_name) for range queries

5. ALERT IF THRESHOLD CROSSED
   → if any metric score crosses threshold for N consecutive traces:
     → print rich alert to stdout
     → append to alerts.jsonl
     → (v3: webhook, slack, discord)

6. VISUALIZE
   spaniq timeline show --metric ResponseDrift --last 200
     → reads timeline SQLite
     → renders matplotlib PNG or rich sparkline to terminal
```

### Key Design Decisions

| Decision | What we chose | What we rejected | Why |
|---|---|---|---|
| Baseline storage | SQLite with `baselines` table | JSONL flat files | Need versioning, prompt_hash lookups, and rollback. SQLite gives this with zero infrastructure. Same choice as DriftGuard V1 |
| Timeline storage | SQLite with `timeline` table | InfluxDB / TimescaleDB | V2 handles <100k data points. SQLite handles millions of rows with proper indexing. No external dependency. Upgrade path to ClickHouse exists for V3 |
| Langfuse integration | Poll Observations API v2 every 30s | Webhooks | Langfuse webhooks are for prompt version events only, not trace events. The Observations API v2 is the recommended data extraction path. Polling with cursor-based pagination is reliable |
| Trace source abstraction | `BaseCollector` ABC with `LangfuseCollector`, `FileCollector`, `SDKCollector` | Single hardcoded source | Users have different observability stacks. The abstraction costs ~30 lines and makes the system composable |
| Monitor loop | `while True` with `time.sleep(interval)` | APScheduler / Celery | Intentionally simple. APScheduler adds dependency weight. A sleep loop is debuggable, killable, and correct for a single-process monitor. DriftGuard V2 used APScheduler and it was over-engineering |
| Alert mechanism | stdout via rich + JSONL append | Slack/Discord webhooks | V2 is a CLI tool, not a service. Stdout is composable with any alerting system via pipe. JSONL is machine-readable for downstream consumers |
| Demo LLM | Groq free tier (Llama 3.3 70B) | OpenAI / Anthropic | 14,400 req/day, 30 RPM, $0. Enough to generate 200 outputs for a demo. OpenAI-SDK-compatible, so the code works with any provider by changing base_url |
| Visualization | matplotlib PNG export + rich sparklines in terminal | Web dashboard | No frontend dependency. PNGs are embeddable in blog posts, READMEs, and X threads. Sparklines work in any terminal. Dashboard is V3 scope |

**Step 0.2 — Design SQLite schemas**

Two tables in a single `spaniq.db` file:

```sql
-- baselines table: stores collections of known-good outputs
CREATE TABLE baselines (
    id          TEXT PRIMARY KEY,           -- uuid4
    name        TEXT NOT NULL,              -- user-friendly name: "refund-v1"
    prompt_hash TEXT NOT NULL,              -- sha256(input_prompt)[:16]
    prompt_text TEXT NOT NULL,              -- the actual prompt
    model_name  TEXT,                       -- "llama-3.3-70b" or null
    outputs     TEXT NOT NULL,              -- JSON array of output strings
    n_outputs   INTEGER NOT NULL,           -- len(outputs), denormalized for queries
    version     INTEGER NOT NULL DEFAULT 1, -- increments on update
    created_at  TEXT NOT NULL,              -- ISO 8601 UTC
    updated_at  TEXT NOT NULL               -- ISO 8601 UTC
);
CREATE INDEX idx_baselines_prompt_hash ON baselines(prompt_hash);
CREATE INDEX idx_baselines_name ON baselines(name);

-- timeline table: time series of metric scores per trace
CREATE TABLE timeline (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    trace_id    TEXT NOT NULL,              -- from collector or auto-generated
    baseline_id TEXT NOT NULL,              -- FK to baselines.id
    metric_name TEXT NOT NULL,              -- "ResponseDriftMetric"
    score       REAL NOT NULL,
    threshold   REAL NOT NULL,
    passed      INTEGER NOT NULL,           -- 0 or 1 (SQLite has no bool)
    timestamp   TEXT NOT NULL,              -- ISO 8601 UTC
    metadata    TEXT                        -- JSON blob for extra context
);
CREATE INDEX idx_timeline_ts ON timeline(timestamp);
CREATE INDEX idx_timeline_metric ON timeline(metric_name, timestamp);
CREATE INDEX idx_timeline_baseline ON timeline(baseline_id, timestamp);
```

Design decisions:
- `outputs` is a JSON array in a TEXT column, not a separate table. At V2 scale (50-200 outputs per baseline), a join table adds query complexity without performance benefit. Denormalized `n_outputs` avoids `json_array_length()` on every query.
- `timeline` uses `TEXT` for timestamps, not `INTEGER` epoch. ISO 8601 strings are human-readable in `sqlite3` CLI, sortable, and parseable by Python's `datetime.fromisoformat()`.
- Composite index on `(metric_name, timestamp)` covers the primary query pattern: "give me the last 200 ResponseDrift scores."

### Phase 1 — Baseline Store (Days 2–3)

**Goal:** Full CRUD for baseline collections. CLI command to collect baselines from any LLM.

**Step 1.1 — `Trace` dataclass (`src/spaniq/monitor/trace.py`)**

```python
    input: str
    output: str

    timestamp: str = field(default_factory=lambda: datetime.now(timezone.utc).isoformat())
    metadata: dict | None = None
```

Minimal. This is the internal representation that all collectors normalize to.

**Step 1.2 — `BaselineStore` (`src/spaniq/monitor/baseline_store.py`)**

```python
class BaselineStore:
    """SQLite-backed storage for baseline output collections."""

    def __init__(self, db_path: str = "spaniq.db"):
        self.db_path = db_path
        self._init_db()

    def create(self, name: str, prompt: str, outputs: list[str],
               model_name: str | None = None) -> str:
        """Store a new baseline. Returns baseline_id."""

    def get(self, baseline_id: str) -> Baseline:
        """Retrieve a baseline by ID."""

    def get_by_name(self, name: str) -> Baseline:
        """Retrieve latest version of a named baseline."""

    def get_by_prompt(self, prompt: str) -> Baseline | None:
        """Look up baseline by prompt hash. Returns None if not found."""

    def update(self, baseline_id: str, new_outputs: list[str]) -> str:
        """Add outputs to existing baseline. Increments version."""

    def list_all(self) -> list[BaselineSummary]:
        """List all baselines with summary info."""

    def delete(self, baseline_id: str) -> None:
        """Delete a baseline and all associated timeline data."""
```

Key implementation details:
- `prompt_hash` is computed as `hashlib.sha256(prompt.encode()).hexdigest()[:16]`
- `outputs` stored as `json.dumps(outputs)`, retrieved with `json.loads()`
- `version` auto-increments on `update()` — old outputs are not deleted, new ones are appended
- Thread-safe: each method opens its own connection with `sqlite3.connect()`. No connection pooling needed at V2 scale
- `_init_db()` uses `CREATE TABLE IF NOT EXISTS` — safe to call multiple times

### Phase 2 — Trace Collectors (Days 4–5)

**Goal:** Three collector implementations that normalize trace data from different sources.

**Step 2.1 — `BaseCollector` ABC (`src/spaniq/monitor/collectors/base.py`)**

```python
from typing import Iterator
from spaniq.monitor.trace import Trace

class BaseCollector(ABC):
    """Abstract base for trace collection from any source."""

    @abstractmethod
    def collect(self) -> Iterator[Trace]:
        """Yield traces from the source. May block waiting for new data."""
        ...

    @abstractmethod
    def name(self) -> str:
        """Human-readable source name for logging."""
        ...
```

**Step 2.2 — `FileCollector` (`src/spaniq/monitor/collectors/file.py`)**

Reads a JSONL file where each line is:
```json
{"input": "...", "output": "...", "timestamp": "...", "trace_id": "..."}
```

Two modes:
- **batch:** read entire file, yield all traces, return
- **tail:** read existing lines, then `poll` for new lines every `interval` seconds (like `tail -f`)

```python
class FileCollector(BaseCollector):
    def __init__(self, path: str, tail: bool = False, interval: float = 1.0):
        self.path = path
        self.tail = tail
        self.interval = interval

    def collect(self) -> Iterator[Trace]:
        with open(self.path) as f:
            for line in f:
                yield self._parse_line(line)
            if self.tail:
                while True:
                    line = f.readline()
                    if line:
                        yield self._parse_line(line)
                    else:
                        time.sleep(self.interval)
```

This is the primary collector for the demos. No external dependency needed.

**Step 2.3 — `LangfuseCollector` (`src/spaniq/monitor/collectors/langfuse.py`)**

Polls the Langfuse Observations API v2 for new GENERATION spans.

```python
class LangfuseCollector(BaseCollector):
    def __init__(self, poll_interval: int = 30):
        self.poll_interval = poll_interval
        self._last_seen: str | None = None  # ISO timestamp cursor

    def collect(self) -> Iterator[Trace]:
        from langfuse import get_client
        client = get_client()

        while True:
            observations = client.api.observations.get_many(
                type="GENERATION",
                fields="core,basic,io",
                limit=100,
                # filter by timestamp > last_seen for cursor pagination
            )
            for obs in observations.data:
                self._last_seen = obs.start_time
                yield Trace(
                    input=obs.input or "",
                    output=obs.output or "",
                    trace_id=obs.trace_id,
                    timestamp=obs.start_time,
                    metadata={"model": obs.model, "langfuse_id": obs.id}
                )
            time.sleep(self.poll_interval)
```

Implementation notes:
- Uses `langfuse` Python SDK v4 (`get_client()` reads `LANGFUSE_PUBLIC_KEY` and `LANGFUSE_SECRET_KEY` from env)
- Cursor-based pagination using `start_time` as the watermark — avoids re-processing old traces
- `fields="core,basic,io"` minimizes payload — only fetches what we need (input, output, model, timestamps)
- `langfuse` is an optional dependency — `ImportError` caught with a helpful message: "pip install spaniq[langfuse]"

**Step 2.4 — `SDKCollector` (`src/spaniq/monitor/collectors/sdk.py`)**

For users who want to integrate directly in Python:

```python
class SDKCollector(BaseCollector):
    """In-process collector for direct SDK usage."""

    def __init__(self):
        self._queue: list[Trace] = []
        self._closed = False

    def ingest(self, input: str, output: str, **kwargs) -> None:
        """Push a trace into the collector."""
        self._queue.append(Trace(input=input, output=output, **kwargs))

    def collect(self) -> Iterator[Trace]:
        while not self._closed or self._queue:
            if self._queue:
                yield self._queue.pop(0)
            else:
                time.sleep(0.1)

    def close(self):
        self._closed = True
```

Usage:
```python
from spaniq.monitor import Monitor, SDKCollector

collector = SDKCollector()
monitor = Monitor(baseline_name="refund-v1", collector=collector)

# in your app code:
collector.ingest(input=user_query, output=llm_response)

# monitor runs in a background thread scoring each ingested trace
```

### Phase 3 — Monitor Loop (Days 6–8)

**Goal:** The core orchestration that ties collector → baseline lookup → evaluate → timeline → alert.

**Step 3.1 — `Monitor` class (`src/spaniq/monitor/monitor.py`)**

```python
class Monitor:
    """Continuous evaluation of LLM traces against stored baselines."""

    def __init__(
        self,
        baseline_name: str,
        collector: BaseCollector,
        metrics: list[BaseMetric] | None = None,
        db_path: str = "spaniq.db",
        alert_after: int = 3,      # alert after N consecutive failures
    ):
        self.baseline_store = BaselineStore(db_path)
        self.timeline_store = TimelineStore(db_path)
        self.alerting = AlertEngine(alert_after=alert_after)
        self.collector = collector
        self.baseline = self.baseline_store.get_by_name(baseline_name)
        self.metrics = metrics or [
            ResponseDriftMetric(),
            SemanticSimilarityMetric(),
            OutputStabilityMetric(),
        ]

    def run(self, max_traces: int | None = None) -> MonitorReport:
        """Run the monitor loop. Blocks until collector is exhausted or max_traces reached."""
        processed = 0
        for trace in self.collector.collect():
            self._process_trace(trace)
            processed += 1
            if max_traces and processed >= max_traces:
                break
        return self._build_report()

    def _process_trace(self, trace: Trace) -> None:
        # 1. construct test case with baselines
        tc = LLMTestCase(
            input=trace.input,
            actual_output=trace.output,
            baseline_outputs=json.loads(self.baseline.outputs),
        )
        # 2. evaluate using V1 metrics
        result = evaluate([tc], self.metrics, verbose=False)

        # 3. store each metric score in timeline
        for mr in result.test_case_results[0].metric_results:
            self.timeline_store.record(
                trace_id=trace.trace_id,
                baseline_id=self.baseline.id,
                metric_name=mr.metric_name,
                score=mr.score,
                threshold=mr.threshold,
                passed=mr.passed,
                timestamp=trace.timestamp,
            )

        # 4. check alert conditions
        self.alerting.check(result.test_case_results[0], trace)
```

Key design decisions:
- `Monitor` owns the full pipeline but delegates to V1's `evaluate()` for scoring — no metric logic duplicated
- `max_traces` parameter allows bounded runs for demos and testing
- `_process_trace` is synchronous and single-threaded — correct at V2 scale. The bottleneck is embedding inference (~100ms/trace), not I/O. Async would add complexity without meaningful throughput gain for <1000 traces/minute
- `MonitorReport` summarizes the run: total traces, pass rate per metric, alerts fired, duration

**Step 3.2 — `TimelineStore` (`src/spaniq/monitor/timeline_store.py`)**

```python
class TimelineStore:
    """SQLite-backed time series of metric scores."""

    def __init__(self, db_path: str = "spaniq.db"):
        self.db_path = db_path
        self._init_db()

    def record(self, trace_id, baseline_id, metric_name, score,
               threshold, passed, timestamp, metadata=None) -> None:
        """Write a single metric score to the timeline."""

    def query(self, metric_name: str, last_n: int = 200,
              since: str | None = None) -> list[TimelineRow]:
        """Retrieve recent scores for a metric. Ordered by timestamp DESC."""

    def summary(self, metric_name: str, last_n: int = 200) -> TimelineSummary:
        """Aggregate stats: mean, std, min, max, pass_rate, trend."""

    def count(self) -> int:
        """Total rows in timeline."""
```

`TimelineSummary` includes:
- `mean_score`, `std_score`, `min_score`, `max_score`
- `pass_rate` (% of traces that passed)
- `trend` — simple linear regression slope over the window. Positive slope on a drift metric = degrading.

**Step 3.3 — `AlertEngine` (`src/spaniq/monitor/alerting.py`)**

```python
class AlertEngine:
    """Fires alerts when metrics cross thresholds for N consecutive traces."""

    def __init__(self, alert_after: int = 3):
        self.alert_after = alert_after
        self._consecutive_failures: dict[str, int] = {}  # metric_name → count
        self.alerts: list[Alert] = []

    def check(self, tc_result: TestCaseResult, trace: Trace) -> None:
        for mr in tc_result.metric_results:
            if not mr.passed:
                self._consecutive_failures[mr.metric_name] = \
                    self._consecutive_failures.get(mr.metric_name, 0) + 1
                if self._consecutive_failures[mr.metric_name] >= self.alert_after:
                    self._fire_alert(mr, trace)
            else:
                self._consecutive_failures[mr.metric_name] = 0

    def _fire_alert(self, mr: MetricResult, trace: Trace) -> None:
        alert = Alert(
            metric_name=mr.metric_name,
            score=mr.score,
            threshold=mr.threshold,
            trace_id=trace.trace_id,
            timestamp=trace.timestamp,
            message=f"ALERT: {mr.metric_name} crossed threshold "
                    f"({mr.score:.4f} vs {mr.threshold}) "
                    f"for {self.alert_after} consecutive traces"
        )
        self.alerts.append(alert)
        # print to stdout via rich
        console.print(f"[bold red]🔴 {alert.message}[/bold red]")
        # append to alerts.jsonl
        self._append_jsonl(alert)
```

Design: consecutive failure window, not sliding window. This prevents noisy one-off failures from triggering alerts while catching sustained degradation. The counter resets on any pass. `alert_after=3` is the default — conservative enough to avoid noise, aggressive enough to catch real drift within minutes.

**Step 4.1 — `visualize.py` (`src/spaniq/monitor/visualize.py`)**

```python
def export_timeline_png(
    timeline_store: TimelineStore,
    metric_name: str,
    output_path: str = "timeline.png",
    last_n: int = 200,
    title: str | None = None,
) -> str:
    """Generate a matplotlib timeline chart as PNG."""
    rows = timeline_store.query(metric_name, last_n=last_n)
    timestamps = [r.timestamp for r in rows]
    scores = [r.score for r in rows]
    passed = [r.passed for r in rows]
    threshold = rows[0].threshold if rows else 0

    fig, ax = plt.subplots(figsize=(12, 4))
    # plot scores as line
    ax.plot(range(len(scores)), scores, color="#2563eb", linewidth=1.5)
    # threshold line
    ax.axhline(y=threshold, color="#dc2626", linestyle="--", linewidth=1, label="threshold")
    # shade failed regions
    for i, p in enumerate(passed):
        if not p:
            ax.axvspan(i - 0.5, i + 0.5, alpha=0.15, color="#dc2626")

    ax.set_title(title or f"{metric_name} — last {last_n} traces")
    ax.set_xlabel("trace #")
    ax.set_ylabel("score")
    ax.legend()
    fig.tight_layout()
    fig.savefig(output_path, dpi=150)
    plt.close(fig)
    return output_path


def print_sparkline(
    timeline_store: TimelineStore,
    metric_name: str,
    last_n: int = 50,
) -> None:
    """Print a rich sparkline of recent scores to terminal."""
    rows = timeline_store.query(metric_name, last_n=last_n)
    scores = [r.score for r in rows]
    blocks = "▁▂▃▄▅▆▇█"
    if not scores:
        return
    mn, mx = min(scores), max(scores)
    rng = mx - mn or 1
    sparkline = "".join(blocks[min(int((s - mn) / rng * 7), 7)] for s in scores)
    console.print(f"[bold]{metric_name}[/bold]: {sparkline}")
```

## Honest Limitations of V2

- **Polling, not streaming.** LangfuseCollector polls every 30s. Real-time detection latency is 30s + scoring time. True streaming would require a webhook or OTEL receiver — that's V3.
- **Single-process, single-thread.** Monitor handles ~10 traces/second (bottlenecked by embedding). Above 600 traces/minute, you need worker processes. At V2 scale this is fine.
- **No automatic baseline refresh.** Baselines are collected once and used forever. In reality, a healthy LLM's output distribution shifts slowly over time. Rolling baseline windows are V3.
- **SQLite write contention.** If the monitor loop writes rapidly while a visualization query runs simultaneously, SQLite's write lock may cause brief delays. WAL mode mitigates this. Not a problem at V2 scale.
- **Alert fatigue.** `alert_after=3` is a fixed window. A smarter system would use exponential backoff or severity-weighted alerting. V3.
- **No multi-model or multi-prompt monitoring.** One monitor instance watches one baseline. Monitoring 10 prompts means running 10 monitors. An orchestration layer is V3.

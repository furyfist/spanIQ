# spanIQ V3 — Build Plan: Component-Level Failure Attribution

> **One-liner:** When a multi-component LLM pipeline degrades, spanIQ V3 tells you WHICH component broke, WHEN exactly it broke, and which downstream failures are cascade — using changepoint detection on metric score series. $0 per diagnosis. Deterministic.

**Prereq:** V1 (4 deterministic metrics, evaluate()) and V2 (Monitor, TimelineStore, BaselineStore, collectors, alerting) are complete — 122 tests passing. V3 layers on V2's TimelineStore exactly the way V2 layered on V1's evaluate(). No V1/V2 code is rewritten.

**Claude Code note:** the full repo is available. Read `src/spaniq/monitor/timeline_store.py`, `src/spaniq/monitor/monitor.py`, and `src/spaniq/core/eval_result.py` before starting Phase 0. The TimelineStore schema is the foundation V3 extends.

## 1. The Problem (and the honest framing)

### 1.1 What the industry cannot do

Multi-agent and multi-component LLM systems fail constantly and nobody can localize the failure. The numbers from the research:

- Multi-agent systems exhibit failure rates of 40–60%; debugging accounts for up to 70% of total maintenance cost (arXiv 2509.08682)
- On the Who&When benchmark (ICML 2025 Spotlight), the best LLM-based attribution method achieves **53.5% accuracy at agent-level and only 14.2% at step-level** — some methods perform below random. Even o1 and DeepSeek R1 fail to reach practical usability
- 65% of enterprise agent failures trace to context drift, not model defects (MemU 2026) — meaning the failure origin is almost never where the symptom appears

Every existing approach (Microsoft AgentRx, AgenTracer, Aegis, causal-inference methods) attacks this by **feeding the failed trace to an LLM and asking it to reason about what went wrong.** Expensive, slow, non-deterministic, and demonstrably unreliable at step-level.

### 1.2 What spanIQ V3 does instead — and the critical scope distinction

There are two different attribution problems. Being precise about which one we solve is what makes this project non-sloppy:

| | Single-trace attribution (Who&When's problem) | Population-level attribution (spanIQ V3) |
|---|---|---|
| Question | "THIS one run failed. Which step caused it?" | "My pipeline degraded across recent traffic. Which COMPONENT broke, and at which trace did it start?" |
| Input | One failed trajectory | Score time series across hundreds of traces |
| Method | LLM reads the transcript and reasons | Changepoint detection on per-component metric series |
| Failure classes covered | Semantic/reasoning errors in one run | Distributional breaks: retrieval failure, model swaps, prompt changes, tool degradation, format breaks |
| Cost | $0.10–1.00 per diagnosis | $0.00 |
| Deterministic | No | Yes |

**spanIQ V3 does NOT compete with Who&When methods on their benchmark.** A single failed trace where an agent made one bad logical decision produces no distributional signal — that genuinely needs an LLM (or a human). What V3 covers is the complementary, arguably more common production class: *systemic* degradation where one component's output distribution shifts and the failure cascades downstream. This distinction goes in the README, the blog post, and every founder conversation. Overclaiming here is the fastest way to lose credibility with the exact people we want to reach.

### 1.3 The core insight

V2 already produces, per trace, per metric, a score written to TimelineStore. A pipeline with N components produces N parallel score series. When the system degrades:

1. **One component's score series shifts first** (the root cause)
2. **Downstream components' series shift later** (the cascade)
3. The time ordering of the shifts IS the attribution

Detecting "where does a time series' statistical regime change" is a 70-year-old solved problem: changepoint detection. CUSUM (Page, 1954) for online detection as traces stream in; PELT (Killick et al., 2012) for exact offline segmentation of a stored timeline. Same statistical literature PSI came from. This is the DriftGuard playbook a third time: take a mature statistical tool from classical monitoring, apply it to LLM telemetry where nobody has.

```
V1 answers: "is this output good?"            (offline eval)
V2 answers: "is my LLM still good?"           (one signal, whole system)
V3 answers: "WHAT broke, WHEN, and what's     (per-component localization
             cascade vs root cause?"           + temporal attribution)
```

## 2. What V3 Ships

1. **Component model** — traces carry typed components (retrieval, tool, generation) aligned with OpenTelemetry GenAI semantic conventions (`gen_ai.operation.name`: `retrieval`, `execute_tool`, `chat`)
2. **PipelineMonitor** — runs V2-style monitoring per component: each component gets its own baseline, rolling window, and V1 metric scores
3. **Changepoint primitives** — hand-implemented online CUSUM + offline PELT (via `ruptures`)
4. **Attributor** — detects changepoints per (component, metric) series, ranks components by earliest break, separates root cause from cascade
5. **Attribution report** — terminal + JSON + PNG: the "who broke first" output
6. **The cascade demo** — multi-component pipeline where retrieval breaks at a known trace and generation degrades downstream; V3 localizes both and ranks them correctly
7. **Validation harness** — synthetic ground-truth injection measuring detection delay and localization error (this is the non-sloppy part)

## 3. What V3 Does NOT Ship

```
❌ Single-trace attribution (LLM-reasoning territory — out of scope, documented)
❌ Who&When benchmark SOTA claims (different problem; optional exploratory study only)
❌ Automatic remediation / self-healing
❌ Web dashboard
❌ Causal graphs between components (V4 candidate; V3 ranks by time ordering only)
❌ Distributed/multi-process monitoring
```

## 4. Architecture

### 4.1 The layering (why this is clean)

```
V1: text → 4 metrics → score                        (pure functions)
V2: trace stream → Monitor → scores → TimelineStore  (state + persistence)
V3: TimelineStore score series → changepoint → rank  (analysis on top of V2's data)
```

V3's input is not raw text. It is the **metric score time series V2 already produces.** The changepoint algorithms never see an LLM output — they see sequences of floats. This means:

- All V1 metrics work per-component with zero modification
- TimelineStore needs one new column (`component`), not a redesign
- The attributor is testable with synthetic float series — no LLM, no fixtures needed for the math layer

### 4.2 Data flow

```
1. INGEST (extended)
   Trace now optionally carries components:
     Trace(
       input, output, trace_id, timestamp,
       components=[
         ComponentSpan(name="retrieval",  kind="retrieval",    output="<retrieved ctx>", latency_ms=120, error=False),
         ComponentSpan(name="search_api", kind="execute_tool", output="<tool result>",   latency_ms=300, error=False),
         ComponentSpan(name="generation", kind="chat",         output="<llm answer>",    latency_ms=900, error=False),
       ]
     )
   Backward compatible: a Trace with components=None behaves exactly as V2
   (treated as a single implicit "default" component).

2. SCORE PER COMPONENT (PipelineMonitor)
   For each ComponentSpan:
     → its own rolling window (deque per component, V2 lesson applied)
     → its own baseline (BaselineStore rows now keyed by component)
     → V1 metrics on the component's text output
     → plus numeric side-series: latency_ms, output_length, error_rate
   Every score written to TimelineStore with component name.

3. DETECT (two modes)
   ONLINE — as each trace arrives:
     CUSUM statistic updated per (component, metric) series.
     S_t = max(0, S_{t-1} + (x_t - μ_baseline - k))
     Alarm when S_t > h. Records alarm trace index.
   OFFLINE — on demand over stored timeline:
     PELT (ruptures, model="rbf") segments each series exactly.
     Returns changepoint trace indices per series.

4. ATTRIBUTE
   Collect all changepoints across (component, metric) pairs.
   Cluster changepoints within a proximity window (default: 10 traces).
   Within a cluster: earliest component = root-cause candidate;
   later components = cascade candidates.
   Confidence = (# of that component's metrics that broke) ×
                (lead time over next component).

5. REPORT
   spaniq attribute --last 500
   ─────────────────────────────────────────────
   degradation event detected (traces 832–960)
     retrieval    break at trace #847  [PSI, cosine]      ← ROOT CAUSE
     generation   break at trace #853  [cosine, JS]         (cascade, +6)
     search_api   no break                                  (healthy)
   verdict: retrieval broke first by 6 traces.
   diagnosis cost: $0.00
```

### 4.3 Key design decisions

| Decision | What we chose | What we rejected | Why |
|---|---|---|---|
| Attribution unit | Component across trace population | Step within one trace | Population-level breaks are what statistics can actually localize. Single-trace reasoning errors have no distributional signal — claiming otherwise would be dishonest. |
| Online detector | Hand-implemented CUSUM (~40 lines) | ruptures online / BOCPD (Adams & MacKay 2007) | ruptures is offline-only by design. BOCPD is Bayesian, heavier, harder to explain in a founder DM. CUSUM is minimax-optimal for detection delay (Moustakides 1986), recursive O(1) per observation, and hand-implementing it shows depth — same reason PSI was hand-implemented in V1. |
| Offline detector | PELT via `ruptures` (BSD license) | Hand-implementing PELT; Binary Segmentation | PELT is exact and O(n) under pruning conditions (Killick 2012). ruptures is the canonical, well-documented implementation. Hand-rolling exact dynamic-programming segmentation adds bug surface, not insight. BinSeg is approximate — no reason to accept approximation at our n (<10k points). |
| PELT cost model | `model="rbf"` (kernel) | `model="l2"` (mean shift only) | Metric score series can shift in variance, not just mean (e.g. ConsistencyMetric under erratic behavior). RBF kernel catches distribution change generally; l2 only catches mean shifts. rbf is the ruptures default for unknown change types. |
| Changepoint input | Metric score series from TimelineStore | Raw text / raw embeddings | Scores are already 1-D, comparable, and stored. Running changepoint on raw embeddings means inventing a multivariate pipeline V2 already solved. Layering > reinventing. |
| Component taxonomy | OTel GenAI semconv operation names (`retrieval`, `execute_tool`, `chat`, `invoke_agent`) | Custom taxonomy | OTel GenAI conventions are becoming the industry standard (v1.41+, Datadog/vendor adoption). Aligning now makes a future OTLP collector a parser, not a redesign. Also a credibility signal to founders. |
| Cascade logic | Time-ordering within proximity cluster | Causal inference (Shapley values, counterfactuals — the arXiv 2509.08682 route) | Temporal precedence is defensible, explainable, and computable at $0. Causal claims require interventions or assumptions we can't validate. Report says "broke first," never "caused" — the verdict language matters. V4 can revisit. |
| CUSUM calibration | Empirical: h set from ARL on normal-trace score series (target ARL₀ ≈ 500 traces between false alarms) | Theoretical ARL integral equations (Page 1954 / Siegmund) | Analytical ARL solutions assume i.i.d. Gaussian increments. Our score series violate i.i.d. (see 4.4). Empirical calibration on real normal series is the honest route — same lesson as V2's threshold recalibration. |
| Timeline schema | Add nullable `component TEXT DEFAULT 'default'` column + index | New table per component | One ALTER TABLE, full V2 backward compatibility, composite index `(component, metric_name, timestamp)` covers the new query pattern. |

### 4.4 The statistical trap to handle explicitly (do not skip)

**Rolling-window scores are autocorrelated.** V2's rolling window PSI means consecutive scores share 19 of 20 window members — the series is smooth by construction. CUSUM's optimality assumes independent increments; feeding it autocorrelated data inflates false alarms (the statistic drifts up on slow trends) or masks true breaks.

Mitigations V3 implements:

1. **Stride sampling for CUSUM input:** update the CUSUM statistic only every `stride` traces (default `stride = window_size // 2 = 10`), so consecutive inputs share at most half their window. This is the standard practical fix (cf. NN-CUSUM literature using non-overlapping mini-batches).
2. **Empirical h calibration:** compute the CUSUM statistic over long synthetic *normal* score series (generated by the validation harness) and set h at the level yielding ARL₀ ≥ 500. Document the calibration script — this is the V3 equivalent of V2's "calibrated thresholds from actual normal-trace PSI measurements."
3. **PELT as the arbiter:** online CUSUM gives the fast alarm; offline PELT on the full series gives the precise break index for the report. The report always quotes PELT's localization, with CUSUM's alarm as detection latency metadata.

This subsection exists because the obvious implementation (CUSUM straight on windowed PSI scores) is subtly wrong — exactly like single-trace PSI was in V2. Finding it before building, instead of after the charts look weird, is the upgrade in process.

### Phase 1 — Changepoint Primitives (Days 2–3)

Pure functions, zero spanIQ imports. Same philosophy as `statistical/` in V1.

**1.1** `cusum.py` — hand-implemented two-sided CUSUM:

```python
@dataclass
class CusumState:
    s_pos: float = 0.0
    s_neg: float = 0.0
    alarm_index: int | None = None
    n_seen: int = 0

def cusum_update(state: CusumState, x: float, mu0: float, k: float, h: float) -> CusumState:
    """One online CUSUM step (Page 1954). Two-sided:
    detects upward shifts (drift scores rising) and downward
    (similarity scores falling). k = slack (typically 0.5σ of
    baseline series), h = decision threshold from calibration.
    Returns new state; state.alarm_index set on first crossing."""
```

Properties to test: no alarm on stationary noise at calibrated h; alarm after injected mean shift; detection delay shrinks as shift magnitude grows; reset-after-alarm behavior; determinism.

**1.2** `pelt.py` — thin `ruptures` wrapper:

```python
def detect_changepoints(series: np.ndarray, penalty: float = 3.0,
                        min_size: int = 10) -> list[int]:
    """Exact offline segmentation via PELT with RBF cost
    (Killick et al. 2012; Truong et al. 2020). Returns
    changepoint indices, excluding the trailing n.
    min_size prevents spurious breaks on tiny segments."""
    algo = rpt.Pelt(model="rbf", min_size=min_size).fit(series.reshape(-1, 1))
    return [bp for bp in algo.predict(pen=penalty) if bp < len(series)]
```

Penalty default 3.0 is a starting point; the validation harness (Phase 6) sweeps it. Document that BIC-style penalty `pen = 3 * log(n)` is the fallback heuristic per ruptures docs.

**1.3** `calibrate.py` — empirical h calibration:

```python
def calibrate_h(normal_series: list[np.ndarray], k: float,
                target_arl0: int = 500) -> float:
    """Run CUSUM over concatenated normal-only series at candidate
    h values; pick smallest h whose mean run length between false
    alarms >= target_arl0. Honest, assumption-free calibration."""
```

Tests: `test_cusum.py` (7), `test_pelt.py` (6: single break localized within ±3, two breaks, no break → empty, min_size respected, determinism, noise robustness), `test_calibrate.py` (3).

Checkpoint commit: `feat(attribution): CUSUM + PELT changepoint primitives with empirical calibration`

### Phase 2 — PipelineMonitor (Days 4–6)

**2.1** `PipelineMonitor` in `attribution/pipeline_monitor.py`. Owns a dict of per-component state — this is V2's hard-won lesson (state lives in the monitor, metrics stay pure) applied at N-component scale:

```python
class PipelineMonitor:
    def __init__(self, pipeline_name: str, collector: BaseCollector,
                 metrics: list[BaseMetric] | None = None,
                 db_path: str = "spaniq.db",
                 window_size: int = 20, cusum_stride: int = 10):
        self._windows: dict[str, deque] = {}           # component → rolling window
        self._cusum: dict[tuple[str, str], CusumState] = {}  # (component, metric) → state
        ...

    def run(self, max_traces=None) -> PipelineReport:
        for trace in self.collector.collect():
            for span in trace.components or [default_span(trace)]:
                self._process_component(trace, span)
```

Per component: maintain window, construct `LLMTestCase` against the component's own baseline, call V1 `evaluate()` directly (NOT via deepcopy-per-call — call metrics the way V2's fixed Monitor does), write scores to TimelineStore with `component=span.name`, update CUSUM every `stride` traces.

**2.2** Numeric side-series: latency_ms and output_length recorded straight into TimelineStore as pseudo-metrics (`latency_ms`, `output_chars`) per component. Free signal — a tool that starts erroring usually shows a latency/length break before a semantic one.

**2.3** CLI: `spaniq pipeline run --name rag-prod --source file --path traces.jsonl`. JSONL format extends V2's with an optional `components` array per line.

Tests: `test_pipeline_monitor.py` (8: per-component windows isolated, scores tagged with component, V2-style trace falls back to default, CUSUM stride respected, online alarm recorded, two components don't share state — the deepcopy-bug regression test at pipeline scale, max_traces, report totals).

### Phase 3 — Attributor (Days 7–8)

**3.1** `attributor.py` — the ranking logic. Input: TimelineStore + component list. Output: `AttributionResult`.

```python
@dataclass
class ComponentBreak:
    component: str
    break_trace_index: int          # PELT localization
    cusum_alarm_index: int | None   # online detection point (latency metadata)
    broken_metrics: list[str]       # which metrics broke at ~this index
    confidence: float               # see formula

@dataclass
class AttributionResult:
    event_window: tuple[int, int]
    root_cause: ComponentBreak | None
    cascade: list[ComponentBreak]
    healthy: list[str]
    verdict: str                    # human sentence, "broke first" language only

def attribute(timeline: TimelineStore, components: list[str],
              metrics: list[str], last_n: int = 500,
              cluster_window: int = 10) -> AttributionResult:
    # 1. pull each (component, metric) series
    # 2. PELT per series → changepoint indices
    # 3. cluster all changepoints within cluster_window traces
    # 4. per cluster: group by component, earliest = root-cause candidate
    # 5. confidence = 0.6 * (broken_metrics / total_metrics)
    #               + 0.4 * min(1, lead_gap / cluster_window)
    # 6. components with no changepoint → healthy
```

Confidence formula rationale: a component where 3 of 4 metrics broke at the same index is a stronger root-cause signal than one metric flickering; a 9-trace lead over the next component is stronger than a 1-trace lead. Weights are heuristic and documented as such — the validation harness reports how often the ranking is correct, which is the number that matters.

**3.2** Edge cases the tests must pin down: simultaneous breaks (gap = 0 → report tie, no root-cause claim), single component (degenerate → V2-equivalent behavior, no attribution language), break only in latency series but not text metrics (still reportable, labeled "operational break"), two independent clusters (two events reported separately).

**3.3** `report.py` — terminal output via rich (the verdict block from §4.2), `--json` for machine consumption, and a stacked PNG: one subplot per component's primary metric series with PELT breaks as vertical lines and the root cause highlighted. That PNG is the V3 chart for the thread.

Tests: `test_attributor.py` (9), `test_attribution_report.py` (4).

Checkpoint commit: `feat(attribution): attributor with changepoint clustering, ranking, cascade labeling`

### Phase 5 — The Cascade Demo (Days 10–11)

The demo that carries the entire V3 story. Three-component simulated RAG pipeline: `retrieval → generation`, with `search_tool` as an independent healthy component.

```
traces   1–100:  all healthy. retrieval returns real context;
                 generation answers grounded.
traces 101–200:  retrieval silently breaks (returns empty/garbage context).
                 generation, fed bad context, starts hedging ~5 traces
                 later as its rolling window fills with degraded outputs.
                 search_tool stays healthy throughout (the control).
```

Built like V2's demos: synthetic deterministic fixtures committed under `demos/fixtures/cascade/` (offline, no API key), optional `--live` flag regenerating via Groq. Expected report:

```
degradation event detected (traces ~101–200)
  retrieval    break at trace ~103  [PSI, cosine, output_chars]   ← ROOT CAUSE
  generation   break at trace ~110  [cosine, JS]                    (cascade, +7)
  search_tool  no break                                             (healthy)
verdict: retrieval broke first by ~7 traces. generation drift is cascade.
diagnosis cost: $0.00
```

The PNG: three stacked series, two vertical break lines visibly offset, one flat healthy series. That offset — root cause leading cascade — is the single image the whole version argues for. Tune fixture severity until the offset is clean, exactly like the V2 chart calibration round.

### Phase 6 — Validation Harness (Days 12–13) — the non-sloppy differentiator

No published spanIQ number without this. `validation/` is a committed, reproducible experiment:

**6.1** `synthetic_series.py`: generate score series mimicking real metric behavior — baseline mean/σ estimated from V2's actual demo timelines, injected mean shifts of {0.5σ, 1σ, 2σ, 4σ} at known indices, autocorrelation matching rolling-window smoothing (AR(1) fit on real series).

**6.2** `run_validation.py` sweeps and reports:

- **Detection delay** (CUSUM): traces between true break and alarm, per shift magnitude
- **Localization error** (PELT): |detected − true| in traces, per magnitude and penalty ∈ {1, 3, 5, 3·log n}
- **False alarm rate**: alarms per 1,000 traces on no-break series at calibrated h
- **Attribution accuracy**: on 200 synthetic two-component cascade scenarios (random lead gaps 2–15), % where the root component is ranked first

# spanIQ V4 Build Plan — Stage 1: Win the Solo Developer

## Version context

```
V1: text → 4 metrics → score                           (pure functions, offline eval)
V2: trace stream → Monitor → scores → TimelineStore    (state + persistence, production monitoring)
V3: TimelineStore score series → changepoint → rank     (per-component attribution)
V4: OTel ingest → all of the above → dashboard + bench  (ecosystem entry + visibility + proof)
```

V4 is the first version where spanIQ becomes something *other people use*, not just something you built. Everything prior was architecture. V4 is distribution.

---

## What V4 Ships

1. **OTelCollector** — OTLP/gRPC + OTLP/HTTP receiver that maps OTel GenAI semantic convention spans to spanIQ `Trace` objects with `ComponentSpan` extraction. Also accepts generic OTel spans with manual `spaniq.*` attributes for non-GenAI instrumented apps.
2. **Streamlit local dashboard** — `spaniq dashboard` command launches a local Streamlit app reading from SQLite. No account, no server, no cloud.
3. **Determinism benchmark** — reproducible benchmark suite comparing spanIQ vs deepeval vs ragas vs promptfoo vs Langfuse evals on cost, variance, and speed.
4. **Quickstart overhaul** — README and docs rewritten around a 2-line "first signal" experience.
5. **pyproject.toml restructured** — optional dependency groups for OTel, dashboard, and benchmark extras.

## What V4 Does NOT Ship

```
❌ Agent-framework-specific integrations (LangGraph/CrewAI/AutoGen) — Stage 2
❌ Langfuse/Phoenix score write-back — Stage 2
❌ Web-hosted dashboard or cloud offering — premature
❌ Auth, RBAC, multi-tenancy — enterprise scope, resist
❌ New statistical detectors — V1-V3 suite is sufficient for Stage 1
❌ DuckDB or alternative storage — SQLite only, keep it simple
```

### OTelCollector design

Two ingestion modes, both producing `Trace` objects with optional `ComponentSpan` lists:

**Mode 1 — GenAI semantic conventions (auto-mapped):**
OTel GenAI spans use standardized attributes (`gen_ai.system`, `gen_ai.operation.name`, `gen_ai.request.model`, etc.). The collector maps:
- `gen_ai.operation.name: "chat"` → `ComponentKind.CHAT`
- `gen_ai.operation.name: "retrieval"` → `ComponentKind.RETRIEVAL`
- `gen_ai.operation.name: "execute_tool"` → `ComponentKind.EXECUTE_TOOL`
- `gen_ai.operation.name: "invoke_agent"` → `ComponentKind.INVOKE_AGENT`
- Span input/output extracted from `gen_ai.content.prompt` / `gen_ai.content.completion` events
- Latency from span duration, error from span status

**Mode 2 — Generic OTel spans (manual tags):**
Any OTel span with `spaniq.input` and `spaniq.output` attributes is accepted. Optional: `spaniq.component` (name), `spaniq.component_kind` (maps to ComponentKind). This lets users instrument *any* app, not just GenAI-convention-compliant ones.

Both modes produce the same `Trace(input, output, components=[...])` that V2/V3 already consume. Zero changes to downstream code.

### Dashboard architecture

Streamlit app with 4 pages, all reading from the existing SQLite TimelineStore:

1. **Overview** — system health summary: total traces, active alerts, last score per metric, health badge per component
2. **Drift Timeline** — interactive time-series charts (Plotly) for each metric, per component. Replaces matplotlib PNGs. Zoom, hover, date-range selection.
3. **Attribution** — V3's changepoint/cascade report rendered as a visual timeline with root-cause and cascade markers. Interactive version of the terminal report.
4. **Alert Log** — sortable/filterable table of all fired alerts with timestamps, metric, component, severity.

No new storage. No new API. Just a read-only Streamlit app pointed at `spaniq.db`.

## Key Design Decisions

| Decision | What we chose | What we rejected | Why |
|---|---|---|---|
| OTel receiver approach | Lightweight OTLP/gRPC + HTTP server embedded in Python process | Full OTel Collector (Go binary) as sidecar | Solo devs won't run a sidecar. An embedded Python gRPC server is `pip install` compatible. Performance ceiling (~500 spans/sec) is fine for individual/small-team scale. |
| GenAI convention version | Pin to semconv v1.41+ development status | Wait for stable | Every competitor (Phoenix, OpenLLMetry, Logfire) ships against development conventions. Waiting means missing the adoption window. Document the pin and commit to updating. |
| Generic span support | Accept any span with `spaniq.*` attributes | GenAI conventions only | Widens the funnel. Users with custom instrumentation (not GenAI-convention-compliant) can still use spanIQ. Costs ~30 lines of mapping code. |
| Dashboard framework | Streamlit | Gradio / custom React / Panel | Streamlit is the Evidently playbook (proven), pip-installable, zero frontend build step, Python-native. Gradio is for ML demos. React requires a build step and JS knowledge. Panel is less adopted. |
| Dashboard storage | Read existing SQLite directly | New API layer / REST server | No new moving parts. Streamlit reads SQLite via the same `TimelineStore` API V2 already provides. The dashboard is a view, not a service. |
| Chart library | Plotly (via streamlit-plotly) | Altair (Streamlit default) / matplotlib | Plotly has better zoom/hover/interactivity for time-series. Altair has a learning curve and limited large-dataset performance. matplotlib is what we're replacing. |
| Benchmark scope | All four competitors + Langfuse | Just deepeval + ragas | "All of the above" per your preference. More comparisons = more credibility. Each runner is ~100 lines. |
| Benchmark dataset | Synthetic JSONL with known ground truth | Real production data | Reproducibility. Anyone can run the benchmark and get the same numbers. No API keys needed for the dataset itself (only for competitor runners). |

**Step 5: Build the GenAI span-to-Trace assembler**

OTel traces arrive as flat lists of spans with parent-child relationships (via `parent_span_id`). The assembler:
1. Groups spans by OTel `trace_id`
2. Identifies the root span (no parent) → becomes the `Trace`
3. Child spans → become `ComponentSpan` entries on that `Trace`
4. Handles out-of-order arrival (buffer spans for up to `assembly_timeout` seconds before flushing incomplete traces)

```python
class TraceAssembler:
    """Assembles flat OTel spans into spanIQ Trace objects with ComponentSpans."""

    def __init__(self, assembly_timeout: float = 5.0):
        self._pending: dict[str, PendingTrace] = {}
        self._timeout = assembly_timeout

    def add_span(self, span: ReadableSpan) -> Trace | None:
        """Add a span. Returns a complete Trace if the trace is ready."""

    def flush_expired(self) -> list[Trace]:
        """Flush traces that have been pending longer than timeout."""
```

Tests: `test_otel_collector.py` — 6 tests: single-span trace, multi-span trace assembly, out-of-order spans, timeout flush, duplicate span handling, empty trace rejection.

Checkpoint commit: `feat(otel): TraceAssembler for multi-span OTel trace reconstruction`

**Step 6: Build the OTLP gRPC + HTTP receiver**

The actual server that listens for incoming OTel spans. Two protocols:
- OTLP/gRPC on port 4317 (standard)
- OTLP/HTTP on port 4318 (standard)

```python
class OTelCollector(BaseCollector):
    """Receives OTel spans via OTLP and converts to spanIQ Traces."""

    def __init__(self, grpc_port: int = 4317, http_port: int = 4318,
                 assembly_timeout: float = 5.0):
        self._converter = SpanConverter()
        self._assembler = TraceAssembler(assembly_timeout)
        self._queue: queue.Queue[Trace] = queue.Queue()

    def start(self) -> None:
        """Start gRPC and HTTP servers in background threads."""

    def collect(self) -> Iterator[Trace]:
        """Yield assembled Traces from the queue. Blocks when empty."""

    def stop(self) -> None:
        """Graceful shutdown of both servers."""
```

Implementation uses `grpcio` for the gRPC server implementing the OTLP `TraceService/Export` RPC, and a lightweight HTTP handler (stdlib `http.server` or `flask` if needed) for the HTTP endpoint.

Tests: 5 integration tests using `opentelemetry-sdk` to send test spans to the collector and verify they arrive as correct `Trace` objects. Test both gRPC and HTTP paths.

Checkpoint commit: `feat(otel): OTLP gRPC + HTTP receiver with BaseCollector interface`

**Step 17: Build the spanIQ benchmark runner**

`benchmarks/runners/spaniq_runner.py`:

```python
def run_spaniq_eval(dataset_path: str, n_runs: int = 5) -> BenchmarkResult:
    """Run spanIQ metrics on the dataset N times, measure variance and time."""
    results = []
    for run in range(n_runs):
        start = time.perf_counter()
        scores = evaluate_dataset(dataset_path)  # uses V1 metrics
        elapsed = time.perf_counter() - start
        results.append(RunResult(scores=scores, time=elapsed, cost=0.0))
    return BenchmarkResult(
        tool="spaniq",
        variance=compute_variance(results),  # should be 0.0
        mean_time=mean([r.time for r in results]),
        total_cost=0.0,
        runs=results
    )
```

The key metric: variance across N identical runs should be **exactly 0.0** for spanIQ (deterministic), while competitors will show non-zero variance.

Checkpoint commit: `feat(benchmark): spanIQ benchmark runner`

- `deepeval_runner.py` — runs G-Eval faithfulness + answer relevancy on the dataset. Requires `OPENAI_API_KEY`. Tracks cost via token counting.
- `ragas_runner.py` — runs faithfulness + context precision. Requires `OPENAI_API_KEY`. Tracks cost.
- `promptfoo_runner.py` — runs via CLI subprocess with YAML config. Uses both deterministic assertions (for fairness) and LLM-graded metrics.
- `langfuse_runner.py` — runs Langfuse eval templates. Requires Langfuse API keys + LLM provider key.

Each runner implements the same interface:
```python
def run_eval(dataset_path: str, n_runs: int = 5) -> BenchmarkResult
```

**Step 19: Build the analysis and reporting module**

`benchmarks/analysis/report.py`:

Generates:
1. **Variance comparison table** — for each tool, show mean score, std dev, min, max, coefficient of variation across N runs. spanIQ's row should show std=0.
2. **Cost comparison table** — total API cost per tool per 100 evaluations. spanIQ = $0.00.
3. **Speed comparison table** — wall-clock time per 100 evaluations.
4. **Reproducibility chart** — scatter plot showing each tool's scores across N runs. spanIQ = flat line, competitors = scattered.

Output formats: terminal table (rich), CSV, PNG chart, and a `summary.md` for the repo.

## Honest Limitations (V4)

- **OTel receiver throughput:** embedded Python gRPC server caps at ~500 spans/sec. Fine for solo devs and small teams; enterprise-scale needs a Go collector sidecar (out of scope).
- **SQLite write contention:** still the scaling bottleneck from V2. Not addressed in V4 (DuckDB is the V5 candidate). At ~10 concurrent writers, expect occasional `database is locked` errors.
- **Streamlit cold start:** first `spaniq dashboard` launch downloads Streamlit's frontend assets (~30s). Subsequent launches are instant. Document this.
- **Benchmark fairness:** competitor runners use their default configs. Power users could tune them for better results. The benchmark measures out-of-the-box experience, not optimal performance.
- **GenAI semconv instability:** conventions are in "Development" status. Attribute names may change. Document the pinned version and commit to updating.
- **Dashboard is read-only:** no configuration, no baseline management, no alert rule editing via the UI. It's a viewer, not a control plane. Control stays in the CLI.

## What Shipped

### `src/spaniq/monitor/`

**`trace.py`** — `Trace` dataclass: `input`, `output`, `trace_id` (auto-UUID), `timestamp` (auto-UTC ISO), `metadata`. Minimal internal representation that all collectors normalize to.

**`baseline_store.py`** — `BaselineStore`: SQLite-backed CRUD for baseline collections. `create`, `get`, `get_by_name`, `get_by_prompt`, `update`, `list_all`, `delete`. Prompt hashed with `sha256[:16]`. Outputs stored as JSON array in TEXT column. Version auto-increments on `update()`, old outputs preserved. Thread-safe: each method opens its own connection.

**`timeline_store.py`** — `TimelineStore`: SQLite time series of metric scores. `record`, `query` (DESC by timestamp), `summary` (mean/std/min/max/pass\_rate/trend). Trend computed as linear regression slope over the query window via `np.polyfit`. Composite index on `(metric_name, timestamp)` covers the primary query pattern.

**`alerting.py`** — `AlertEngine`: consecutive-failure window alerting. Counter per metric increments on failure, resets on pass. Alert fires when count reaches `alert_after` (default 3). Each alert printed to stdout via rich and appended to `alerts.jsonl`.

**`monitor.py`** — `Monitor`: orchestrates the full pipeline. Owns `BaselineStore`, `TimelineStore`, `AlertEngine`, and the `BaseCollector`. For each trace: constructs `LLMTestCase` with baseline outputs → calls V1 `evaluate()` → records each `MetricResult` to timeline → passes results to `AlertEngine`. Returns `MonitorReport` with total traces, duration, alerts fired, and per-metric pass rates.

**`visualize.py`** — `export_timeline_png()`: matplotlib line chart with threshold line and failed-region shading, saved as PNG at 150 DPI. `print_sparkline()`: unicode block chars rendered via rich, with pass rate and threshold in dim text.

**`collectors/base.py`** — `BaseCollector` ABC: `collect() -> Iterator[Trace]`, `name() -> str`.

**`collectors/file.py`** — `FileCollector`: reads JSONL files in batch or tail mode. Skips malformed lines with `logging.warning`. Handles missing `input`/`output` fields gracefully.

**`collectors/sdk.py`** — `SDKCollector`: in-process queue. `ingest(input, output, **kwargs)` pushes traces; `collect()` yields them; `close()` signals no more input.

**`collectors/langfuse.py`** — `LangfuseCollector`: polls Langfuse Observations API v2 (langfuse SDK v4) for `GENERATION` spans. Cursor pagination using `from_start_time` watermark. Lazy import with helpful `ImportError` message for missing optional dependency.

### CLI (`src/spaniq/cli/runner.py`)

Refactored from raw `sys.argv` to `argparse`. New subcommands:

```
spaniq baseline collect  --name --prompt [--source groq|file] [--n] [--model] [--path]
spaniq baseline list

spaniq timeline show     --metric [--last]
spaniq timeline export   --metric [--last] [--output]

spaniq demo prompt-injection  [--offline]
spaniq demo model-swap        [--offline]

```

### Demos (`src/spaniq/demos/`)

Three reproducible demos, each with a `run(offline=bool)` API and a `if __name__ == "__main__": run()` entrypoint.

**`prompt_injection.py`**: 20 baseline customer-support outputs → 20 normal traces → 20 pirate-persona injected traces. `ResponseDriftMetric` PSI spikes at ~trace 21 as pirate vocabulary diverges. Alert fires at trace 23.

**`model_swap.py`**: 20 baseline outputs from `llama-3.3-70b-versatile` on coding questions → 20 more 70B traces → 20 from `llama-3.1-8b-instant` (shorter, sparser). `OutputStabilityMetric` JS divergence detects structural change.

**`rag_breakage.py`**: 20 baseline outputs with retrieval context (policy text in system prompt) → 20 with-context traces → 20 without-context traces (LLM hedges: "I'm not sure", "generally"). `SemanticSimilarityMetric` drops; `ResponseDriftMetric` detects hedging vocabulary.

All three have pre-generated JSONL fixtures in `src/spaniq/demos/fixtures/` — demos run fully offline with `--offline` flag.

## Key Design Decisions (V2)

| Decision | What we chose | Why |
|---|---|---|
| Baseline storage | SQLite `TEXT` column for JSON outputs | Avoids join table at V2 scale (20-200 outputs). Denormalized `n_outputs` avoids `json_array_length()` on queries |
| Timeline timestamps | ISO 8601 `TEXT`, not `INTEGER` epoch | Human-readable in `sqlite3` CLI, sortable as string, parseable with `fromisoformat()` |
| Monitor loop | `while True` + `time.sleep` | Debuggable and killable. APScheduler adds dependency weight without benefit at V2 scale |
| Alert mechanism | Consecutive-failure window | Prevents noisy one-off failures from alerting. Resets on any pass — catches sustained degradation, not noise |
| Langfuse integration | Poll Observations API v2, `from_start_time` cursor | Langfuse webhooks are for prompt version events, not trace events. Cursor pagination is reliable at 100 obs/request |
| Groq integration | `groq` SDK directly | Simpler than the openai SDK with custom `base_url`. Purpose-built client for Groq API |
| Visualization backend | matplotlib `Agg` backend | Non-interactive, safe in CLI and test environments. No display required |

## Deviations from Plan

**`__init__.py` lazy imports**: The monitor package's `__init__.py` uses `__getattr__` for deferred imports of `Monitor`, `TimelineStore`, and `SDKCollector`. Required because these modules depend on each other and on modules not yet written during early test phases. Ensures `from spaniq.monitor.trace import Trace` works before `Monitor` exists.

**Synthetic fixtures**: Phase 5.6 required a `GROQ_API_KEY` to pre-generate fixtures. Built deterministic synthetic fixtures instead that produce the same statistical signal — pirate vocabulary for prompt injection, short terse responses for model swap, hedging phrases for RAG breakage.

**`AlertEngine` as dataclass**: Plan showed it as a plain class. Implemented as a `@dataclass` with `__post_init__` for `_consecutive` counter initialization. Cleaner construction and no-arg usage in `Monitor`.

**CLI refactor**: Plan showed adding V2 subcommands to the existing raw `sys.argv` runner. Refactored to `argparse` (Phase 1.3) before adding any V2 commands — necessary for maintainability with 4 top-level command groups.

## Honest Limitations

- **Polling, not streaming.** `LangfuseCollector` polls every 30s. Real-time latency = poll interval + scoring time.
- **Single-process.** Monitor handles ~10 traces/second (embedding bottleneck). Above 600 traces/min, worker processes needed.
- **No automatic baseline refresh.** Baselines are static. Rolling windows are V3.
- **SQLite write contention.** WAL mode not explicitly enabled. Not a problem at V2 scale but may cause brief delays under concurrent write + read.
- **Alert fatigue.** `alert_after=3` fixed window. Adaptive backoff is V3.
- **One monitor per baseline.** Monitoring 10 prompts means 10 Monitor instances. Orchestration layer is V3.

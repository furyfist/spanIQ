# spanIQ V1 — Completion Report

## TL;DR

- 4 deterministic metrics shipped: `ResponseDriftMetric` (PSI), `SemanticSimilarityMetric` (cosine), `OutputStabilityMetric` (JS divergence), `ConsistencyMetric` (KS test)
- 71 tests, 0 failures, 0 lint errors across 13 test files
- $0.00 per eval run — 0 API calls, all computation is local
- 1,000 test cases × 3 metrics costs ~$34.20 with LLM-as-judge; costs $0.00 with spanIQ
- Embedding inference (MiniLM, 384 dimensions) is the dominant runtime cost — a 1-case quickstart run takes ~10s on first call (model load), ~4s on warm runs

---

## Problem Statement

Every mainstream LLM evaluation framework (deepeval, ragas, braintrust) calls a frontier LLM to score outputs. The cost compounds: N test cases × N metrics × 3 LLM calls per metric judgment. At GPT-4o pricing (≈$0.0025/1K input tokens, ≈$0.01/1K output tokens, ~800 tokens per judge call), 1,000 test cases × 3 metrics = 9,000 API calls = ~$34.20 per run. Running this in daily CI costs ~$1,000/month before you have any meaningful test suite size.

The second problem is that LLM judges are non-deterministic. The same input produces a different score across runs because temperature > 0 is the default. This makes regression detection unreliable: you cannot distinguish a genuine score drop from judge noise. You cannot gate a CI pipeline on a signal that changes between identical runs.

## What Shipped

### Core Components

**`LLMTestCase` — `src/spaniq/core/test_case.py`**

A dataclass with six fields: `input` (str), `actual_output` (str), `expected_output` (str | None), `baseline_outputs` (list[str]), `retrieval_context` (list[str] | None), `metadata` (dict | None). `__post_init__` raises `ValueError` on empty `input` or `actual_output`. `baseline_outputs` defaults to an empty list rather than being required — this allows constructing a test case without baselines when only `SemanticSimilarityMetric` with `expected_output` is used.

The key decision: `baseline_outputs` is a first-class field on the test case, not a parameter passed to each metric. This means one test case definition works across all metrics that need a reference distribution. The alternative — passing baselines per-metric — would force duplication and allow accidental mismatches between the baselines a metric sees and those another metric sees on the same case.

**`EvalResult` and related types — `src/spaniq/core/eval_result.py`**

Four dataclasses: `CostReport` (n_test_cases, n_metrics, spaniq_cost, llm_judge_cost, savings property), `MetricResult` (metric_name, score, threshold, passed, reason, details), `TestCaseResult` (test_case, metric_results, passed property — `all(mr.passed for mr in self.metric_results)`), `EvalResult` (test_case_results, total_passed, total_failed, total_cases, cost_report, duration_seconds).

`TestCaseResult.passed` is a computed property, not a stored field — it cannot drift out of sync with the individual metric results that underlie it.

**`evaluate()` — `src/spaniq/core/evaluate.py`**

Signature: `evaluate(test_cases, metrics, max_workers=4, verbose=True) -> EvalResult`

Uses `ThreadPoolExecutor` with `copy.deepcopy(metric)` per submission to avoid shared mutable state across threads. Each `(test_case, metric)` pair is submitted as an independent future. Results are collected per test case by pre-grouping futures in a dict keyed by `id(tc)` before the executor closes. Wall time is measured with `time.perf_counter()`.

The threading choice (not multiprocessing): numpy and scipy release the GIL during numerical computation, so threads give real parallelism for the statistical metrics. For embedding inference, `sentence-transformers` holds the GIL more tightly — a `ProcessPoolExecutor` would be faster for large datasets but adds serialization overhead and startup cost. At V1 scale this tradeoff favors threads.

**`BaseMetric` — `src/spaniq/metrics/base.py`**

Abstract class with `threshold` and `name` in `__init__`, abstract `measure(test_case) -> float` and `_check_threshold(score) -> bool`, and concrete `is_successful()` that calls `_check_threshold(self.score)` after asserting score is not None. The `name` defaults to `self.__class__.__name__` — this is what appears in `MetricResult.metric_name` and in `AssertionError` messages from `assert_eval`.

**`assert_eval()` — `src/spaniq/__init__.py`**

Calls `evaluate([test_case], metrics, verbose=False)`, inspects the first `TestCaseResult`, and raises `AssertionError` with a message of the form `"Metrics failed: MetricName: 0.1234 (threshold: 0.05)"` if any metric failed. Designed for use directly inside pytest test functions as a drop-in for `deepeval.assert_test`.

**`cost_report.py` — `src/spaniq/reporting/cost_report.py`**

Constants: `_GPT4O_INPUT_PER_1K = 0.0025`, `_GPT4O_OUTPUT_PER_1K = 0.01`, `_AVG_TOKENS_PER_JUDGE_CALL = 800`, `_CALLS_PER_METRIC = 3`. Formula: `total_calls = n_test_cases × n_metrics × 3`, `total_tokens = total_calls × 800`, `input_cost = (tokens × 0.6) / 1000 × 0.0025`, `output_cost = (tokens × 0.4) / 1000 × 0.01`. Result rounded to 4 decimal places. Uses `rich` for terminal table output.

**`cli/runner.py` — `src/spaniq/cli/runner.py`**

`main()` checks `sys.argv` for `test run` prefix, then calls `subprocess.call([sys.executable, "-m", "pytest", *remaining_args, "-v"])`. Registered as the `spaniq` entry point in `pyproject.toml`. This is intentionally minimal — it wraps pytest rather than reimplementing test discovery.

### Metrics

**`ResponseDriftMetric`**

Measures whether the word-frequency distribution of `actual_output` has shifted from the aggregate distribution of `baseline_outputs`. Uses PSI (Population Stability Index). `baseline_outputs` are concatenated into a single string, tokenized by whitespace after `.lower()`, and converted to a frequency vector over the union vocabulary with `actual_output`. PSI is computed between these two vectors. Default threshold: `0.10`. Pass condition: `score < threshold` (lower PSI = less drift). Raises `ValueError` if `baseline_outputs` is empty.

**`SemanticSimilarityMetric`**

Measures cosine similarity between the embedding of `actual_output` and either `expected_output` (preferred) or the mean of `baseline_outputs` embeddings. When using baseline mean, the mean vector is re-normalized before computing dot product. Default threshold: `0.70`. Pass condition: `score >= threshold` (higher similarity = pass). Raises `ValueError` if neither `expected_output` nor `baseline_outputs` is provided.

**`OutputStabilityMetric`**

Measures structural consistency of `actual_output` against the distribution of structural features across `baseline_outputs`. Extracts four features per text: character count (`len(text)`), word count (`len(text.split())`), sentence count (split on `.`, filter empty), average word length (`np.mean([len(w) for w in words])`). Computes JS divergence per feature between the baseline column distribution and the single current value. If all baselines have the same value for a feature, the fallback is `0.0` if current matches, `1.0` if it doesn't (JS is undefined for a degenerate distribution). Final score is `np.mean` across the four feature JS scores. Default threshold: `0.15`. Pass condition: `score < threshold`.

**`ConsistencyMetric`**

Measures whether `actual_output` falls within the normal variation of `baseline_outputs` in embedding space. Embeds all texts together, computes cosine distances from `actual_output` to each baseline (actual-to-baseline distances), and cosine distances between all baseline pairs (baseline-to-baseline distances, upper triangle only). Runs a two-sample KS test between these two distance arrays. The KS statistic becomes the score. Default threshold: `0.05`. Pass condition: `score < threshold` (low KS statistic = actual output is not distinguishable from baseline variation). Requires minimum 5 `baseline_outputs` — enforced with `ValueError`.

### Statistical Primitives

**PSI — `src/spaniq/statistical/psi.py`**

`compute_psi(baseline: np.ndarray, current: np.ndarray, n_bins: int = 10) -> float`

Formula: `Σ (current_pct - baseline_pct) × ln(current_pct / baseline_pct)` over bins. Non-obvious decisions: (1) bin edges are computed from `np.percentile(baseline, ...)` only — not from the combined distribution. This is correct statistical practice: the reference distribution defines the bins. (2) `np.unique()` is called on bin edges to remove duplicates that arise when the baseline has repeated values (e.g., a constant or sparse distribution). Without this, `np.histogram` raises on duplicate edges. (3) Epsilon smoothing `1e-6` is added to both counts before normalizing to prevent `log(0)`.

**KS test — `src/spaniq/statistical/ks.py`**

`compute_ks(baseline: np.ndarray, current: np.ndarray, alpha: float = 0.05) -> KSResult`

Wraps `scipy.stats.ks_2samp`. Returns a `KSResult` NamedTuple with `statistic: float`, `p_value: float`, `significant: bool`. Non-obvious decision: `significant` is cast with `bool(result.pvalue < alpha)` — explicit Python `bool` cast, not `np.bool_`. `scipy.stats.ks_2samp` returns `np.bool_` for the comparison, which fails `type(x) is bool` checks. This is tested explicitly in `test_ks.py::test_significant_is_python_bool`.

**JS divergence — `src/spaniq/statistical/js.py`**

`compute_js(baseline: np.ndarray, current: np.ndarray, n_bins: int = 50) -> float`

Wraps `scipy.spatial.distance.jensenshannon`, which returns JS distance (square root of JS divergence), bounded [0, 1]. Non-obvious decisions: (1) shared bin edges computed with `np.linspace(combined_min, combined_max, n_bins + 1)` across both arrays — this is required for the histograms to be comparable. PSI uses baseline-only edges; JS uses shared edges because JS is a symmetric measure and neither distribution is the reference. (2) Early return `0.0` when `combined_min == combined_max` — prevents `np.linspace` from generating a degenerate single-value range. (3) `n_bins=50` vs PSI's `n_bins=10` — JS is used for structural feature distributions which tend to be more granular.

**Embeddings — `src/spaniq/statistical/embeddings.py`**

`embed(texts: list[str]) -> np.ndarray` returns shape `(n, 384)` float32, normalized. `cosine_similarity(a, b) -> float` is `float(np.dot(a, b))` — works because embeddings are normalized at encode time, making dot product equal to cosine similarity. Model: `sentence-transformers/all-MiniLM-L6-v2`. Non-obvious decision: lazy loading via module-level `_model = None` global. `SentenceTransformer` is imported and instantiated only on the first call to `embed()`. This means importing spaniq or creating metrics has zero cost — the 80MB model download and ~2s load time only happens when embedding is actually invoked.

│   ├── evaluate.py          # evaluate() — orchestrates metrics via ThreadPoolExecutor
│   └── test_case.py         # LLMTestCase dataclass
├── metrics/
│   ├── __init__.py          # re-exports all 4 metrics
│   ├── base.py              # BaseMetric abstract class
│   ├── consistency.py       # KS test on embedding distance distributions
│   ├── output_stability.py  # JS divergence on structural features
│   ├── response_drift.py    # PSI on word-frequency distributions
│   └── semantic_similarity.py  # cosine similarity via MiniLM embeddings
├── reporting/
│   ├── __init__.py
│   └── cost_report.py       # cost estimation formula + rich terminal output
└── statistical/
    ├── __init__.py
    ├── embeddings.py        # lazy-loaded MiniLM wrapper
    ├── js.py                # JS distance via scipy.spatial.distance.jensenshannon
    ├── ks.py                # KS test via scipy.stats.ks_2samp
    └── psi.py               # PSI — hand-implemented with numpy
```

### Data Flow

```
User constructs:
  LLMTestCase(input, actual_output, baseline_outputs, ...)

Passed to:
  evaluate(test_cases, metrics, max_workers=4)
    |
    ├── copy.deepcopy(metric) for each (test_case, metric) pair
    ├── ThreadPoolExecutor.submit(_run_single, metric_copy, test_case)
    |
    └── _run_single(metric, test_case):
          metric.measure(test_case)
            |
            ├── ResponseDriftMetric  → _token_frequencies() → compute_psi()
            ├── SemanticSimilarityMetric → embed() → cosine_similarity()
            ├── OutputStabilityMetric → _extract_features() → compute_js()
            └── ConsistencyMetric → embed() → cosine distances → compute_ks()
            |
            returns float score, sets metric.score and metric.reason
          |
          returns MetricResult(metric_name, score, threshold, passed, reason)

    collect futures → list[TestCaseResult]
    build_cost_report(n_test_cases, n_metrics) → CostReport

Returns:
  EvalResult(test_case_results, total_passed, total_failed, total_cases,
             cost_report, duration_seconds)
    |
    if verbose: print_report(eval_result) via rich
```

### Key Design Decisions

| Decision | What we chose | What we rejected | Why |
|---|---|---|---|
| Embedding model | `all-MiniLM-L6-v2` (384 dims, 80MB) | Larger models (MPNet, E5) | Runs on CPU in ~100ms per sentence. Sufficient for similarity detection, not text generation. Keeps the install size manageable. |
| Parallelism | `ThreadPoolExecutor` (threads) | `ProcessPoolExecutor` (processes) | scipy/numpy release the GIL so threads get real concurrency for statistical metrics. Processes add serialization cost and startup overhead that outweighs gains at V1 scale. |
| Metric state isolation | `copy.deepcopy(metric)` per future | Shared metric instance | `BaseMetric` stores `self.score` and `self.reason` as mutable instance state. Without deepcopy, concurrent calls corrupt each other's state. |
| `baseline_outputs` placement | Field on `LLMTestCase` | Parameter on each `metric.measure()` | One definition of "what normal looks like" per test case, shared across all metrics. Avoids accidental inconsistency where different metrics see different baselines for the same case. |
| Determinism guarantee | All randomness seeded via `np.random.default_rng` in tests, no randomness in production code | Stochastic scoring | Every production code path is deterministic: PSI/KS/JS are pure functions of their inputs, embeddings are deterministic for fixed weights. Tests assert exact float equality across two calls. |
| `significant` type in KSResult | Explicit `bool()` cast | Raw numpy comparison result | `scipy.stats.ks_2samp` returns `np.bool_` for comparisons. `type(x) is bool` would fail. Explicit cast documented and tested in `test_ks.py::test_significant_is_python_bool`. |
| Pass/fail for `TestCaseResult` | Computed property (`all(mr.passed ...)`) | Stored field set at construction | A stored boolean could drift out of sync if `metric_results` were mutated. The property is always consistent with its source data. |

## Honest Limitations

**No subjective quality evaluation.** spanIQ cannot answer "is this response helpful?" or "is this a good answer?". PSI and cosine similarity measure distributional similarity, not quality. A response that is confidently wrong but stylistically similar to baselines will pass every spanIQ metric.

**No factual accuracy or hallucination detection.** `ResponseDriftMetric` detects vocabulary shift; it has no knowledge of whether a claim is true. A hallucinated but fluent response that uses similar vocabulary to baselines will pass.

**Requires baselines — no zero-shot evaluation.** Every metric except `SemanticSimilarityMetric` with `expected_output` requires `baseline_outputs`. If you have no prior outputs from your LLM in a known-good state, you cannot use spanIQ. The framework provides no tooling for collecting or managing baselines.

**No safety or toxicity detection.** These require either a dedicated safety classifier or LLM judgment. JS divergence on structural features will not detect toxic content.

**`OutputStabilityMetric` degrades with small baselines.** With fewer than ~10 baseline outputs, each structural feature column has too few values for a meaningful JS divergence estimate. The fallback for a degenerate all-same-value column (hard 0.0 or 1.0) is rough. This metric is most meaningful with 20+ baselines.

**`ConsistencyMetric` requires exactly ≥5 baselines, but is more reliable with 15+.** The baseline-to-baseline distance distribution (upper triangle of the pairwise matrix) has `n*(n-1)/2` values. At n=5, that is 10 values — barely enough for a KS test. At n=10 it is 45 values, which is more reliable.

**No production monitoring.** V1 is an offline evaluation framework. There is no mechanism to run metrics on live traces, emit results to a time-series store, or alert on drift in production.

**No streaming or incremental evaluation.** `evaluate()` blocks until all test cases and metrics are complete. For large test suites, there is no progress reporting beyond the final summary.

**Thread safety of the embedding model.** The `_model` global in `embeddings.py` is written once on first call without a lock. If two threads call `embed()` simultaneously on a cold start, both will attempt to load the model. In practice this is harmless (the second write is idempotent and the model object is thread-safe), but there is a brief window of redundant work.

## Deviations from Plan

**Phase 0 — Scaffolding:** `uv` was not installed and the user preferred venv. Replaced `uv init`, `uv sync`, and `uv run` with `python -m venv .venv` and `.venv/Scripts/pip install -e ".[dev]"`. The CI workflow was updated to use `pip` instead of `uv`. Everything else shipped as planned.

**Phase 1 — Statistical Core:** Shipped as planned. PSI, KS, JS, and the embedding wrapper match the build plan's specified function signatures and implementation decisions.

**Phase 2 — Data Model:** Shipped as planned.

**Phase 3 — Metrics:** `OutputStabilityMetric` — the build plan described JS divergence on structural features but did not specify the degenerate-distribution fallback (`0.0` or `1.0` when all baseline values are identical for a feature). This was added during implementation. One test was revised: the planned `test_similar_length_passes` was replaced with `test_score_is_float_in_valid_range` after discovering that JS with 4 baselines and a single current value produces the same score regardless of position within the baseline range — the test was asserting behavior the metric cannot guarantee at that sample size.

**Phase 4 — Evaluate Pipeline:** The `tc_futures` dict uses `id(tc)` as key. This assumes test cases are not duplicated objects — a user passing the same `LLMTestCase` instance twice would produce incorrect grouping. This was not in the plan and is a latent bug for an edge case that does not arise in normal use.

**Phase 5 — CLI:** Shipped as planned. The CLI is intentionally thin — it delegates entirely to pytest.

**Phases 6, 7, 8 — Examples, Docs, Polish:** Shipped as planned. The `v0.1.0` tag was created locally. PyPI publishing was deferred pending community feedback, as stated in the plan.

```
total_calls  = n_test_cases × n_metrics × 3
total_tokens = total_calls × 800
input_cost   = (total_tokens × 0.6) / 1000 × 0.0025
output_cost  = (total_tokens × 0.4) / 1000 × 0.01
total_cost   = round(input_cost + output_cost, 4)

## What V1 Does Not Solve (V2 Scope)

- **Baseline collection tooling.** Users must build `baseline_outputs` manually by running their LLM and saving outputs. There is no CLI command, SDK method, or integration to collect baselines from production logs, existing test suites, or LangSmith/Langfuse traces.

- **Parallel eval uses `ThreadPoolExecutor` but embedding inference is CPU-bound.** For workloads dominated by `SemanticSimilarityMetric` or `ConsistencyMetric`, a `ProcessPoolExecutor` with serialized model instances would be faster. The current implementation does not let the user choose.

- **No production monitoring path.** There is no mechanism to run metrics continuously against live LLM traces. V1 is entirely batch/offline. A production monitor would need to consume an event stream, maintain rolling baseline windows, and emit scores to a time-series backend.

- **`TestCaseResult.passed` uses `all()` — no partial pass logic.** There is no way to express "must pass at least 2 of 3 metrics" or weight metrics differently. All metrics are equally required.

- **No threshold auto-calibration.** Default thresholds (PSI < 0.10, cosine >= 0.70, JS < 0.15, KS < 0.05) were chosen from industry convention and informal testing. There is no tooling to calibrate thresholds against a labeled dataset of pass/fail examples for a specific use case.

- **`OutputStabilityMetric` uses only 4 structural features.** Sentence count splits on `.` only, which miscounts at abbreviations, decimal numbers, and mid-sentence ellipsis. Adding tokenizer-aware sentence segmentation (e.g., via `nltk.sent_tokenize`) would improve accuracy at the cost of an additional dependency.

- **No RAG-specific metrics.** `retrieval_context` is a field on `LLMTestCase` but no metric uses it in V1. Context precision, context recall, and faithfulness relative to retrieved documents are deferred.

- **No agent-specific metrics.** Tool call sequences, planning steps, and multi-turn consistency are out of scope for V1.

- **The `_model` global in `embeddings.py` has no cleanup path.** For long-running processes or test suites, there is no way to unload the model from memory. This is fine for CLI use but matters for server deployments.

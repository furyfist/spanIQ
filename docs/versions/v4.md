## Shipped capabilities

### OTelCollector ŌĆö ecosystem entry

OTLP/gRPC (port 4317) and OTLP/HTTP (port 4318) receiver that maps OTel spans to
spanIQ `Trace` objects with `ComponentSpan` extraction. Two ingestion modes:

- **GenAI semantic conventions (auto-mapped):** `gen_ai.operation.name` maps to
  `ComponentKind` (chat / retrieval / execute_tool / invoke_agent); input/output
  from prompt/completion events; latency from span duration; error from status.
- **Generic spans (manual tags):** any span with `spaniq.input` / `spaniq.output`
  attributes is accepted, with optional `spaniq.component` / `spaniq.component_kind`.
  Lets users instrument any app, not just GenAI-convention-compliant ones.

Both modes produce the same `Trace(input, output, components=[...])` that V2/V3
already consume. The middle pipeline is untouched.

`spaniq collect-otel --baseline my_baseline` starts the receiver and pipes traces
into the Monitor.

### Streamlit local dashboard ŌĆö visibility

`spaniq dashboard` launches a local Streamlit app reading directly from SQLite. No
account, no server, no cloud. Four pages:

1. **Overview** ŌĆö total traces, active alerts, per-component health badge, last
   score per metric, ingestion sparkline.
2. **Drift Timeline** ŌĆö interactive Plotly time-series per metric per component,
   with threshold lines. Replaces the matplotlib PNGs.
3. **Attribution** ŌĆö V3's changepoint/cascade report as an interactive cascade
   visualization with root-cause and cascade markers.
4. **Alert Log** ŌĆö sortable/filterable table of fired alerts, CSV export.

Read-only viewer, not a control plane. Control stays in the CLI.

### Determinism benchmark ŌĆö proof

Reproducible benchmark suite comparing spanIQ against LLM-as-judge competitors on
variance, cost, and speed across three committed JSONL datasets (qa_factual,
summarization, rag_retrieval). The headline: spanIQ's std dev is exactly 0.0000
across N identical runs; every LLM judge shows non-zero spread.

Runners share one interface and return the same `BenchmarkResult`. Competitors run
on the free Groq API for fairness, with a token-based production-cost estimate
reported so the table reflects real spend on a paid model.

```bash
spaniq benchmark --tool spaniq --runs 5
spaniq benchmark --tool spaniq,groq,deepeval,ragas,langfuse --runs 3
```

## Benchmark runners ŌĆö completion (this cycle)

The V4 release shipped the benchmark scaffold with three working runners (spaniq,
groq, deepeval). This cycle completed the two remaining competitors so the
determinism table covers the full planned set of five tools.

### ragas runner ŌĆö rewritten to the v0.4 collections API

The legacy `from ragas.metrics import faithfulness` runner was replaced with the
v0.4 collections API (`SingleTurnSample`, `EvaluationDataset`, `Faithfulness` from
`ragas.metrics.collections`), scoring each sample with `single_turn_ascore`.

- Judged by Groq's LLaMA via the OpenAI-compatible endpoint through `llm_factory`,
  with a LangChain `ChatOpenAI` wrapper fallback if the v0.4 factory rejects a
  non-OpenAI client.
- Runs on `rag_retrieval.jsonl` only (faithfulness needs retrieved context); guards
  against datasets that lack context.
- Token-based production-cost estimate.
- Graceful skip (raises `ImportError` / `EnvironmentError` / `ValueError`, caught by
  the CLI) when ragas or the Groq key are missing.

### langfuse runner ŌĆö Langfuse-style LLM-as-judge

Langfuse has no standalone local eval function ŌĆö its LLM-as-a-Judge runs server-side
on ingested traces. This runner replicates the methodology, not the platform: the
same kind of judge prompt template a Langfuse user configures, run against Groq
locally. No Langfuse SDK or server required.

- JSON `{score, reasoning}` 1-10 template, normalized to 0-1, with loose-number
  fallback parsing.
- Concurrent batch judging via `asyncio.gather`.
- Real token-usage cost from the judge calls.
- Runs on `qa_factual.jsonl` for apples-to-apples LLM-judge variance comparison.

### Shared and supporting changes

## Post-release fixes (found during full-suite validation)

Two pre-existing test failures were diagnosed and fixed. No API changes; both fixes
are backwards compatible.

- **ResponseDriftMetric PSI scoring** ŌĆö `_discrete_psi` used `eps=1e-6` smoothing
  that made tokens missing from one distribution produce exploding `log(c/b)`
  spikes, so even an output identical to a baseline scored as drift (PSI 9.8).
  Replaced with proper Laplace (add-1) smoothing. Separately, `measure()` compared
  the output against the merged baseline corpus, inflating drift via vocabulary-size
  mismatch; it now takes the minimum PSI across individual baselines ŌĆö drift is
  distance from the nearest known-good response. Identical output now scores 0.0
  (pass), clearly different output 0.43 (fail).

- **TimelineStore query ordering** ŌĆö `query()` ran `ORDER BY timestamp DESC` then
  reversed the result, contradicting its own SQL, docstring, and the test. It now
  honors the DESC (newest-first) contract by default, with an explicit
  `ascending=True` flag for chronological consumers. All call sites that plot or
  compute trends over time (visualizations, sparkline, `query_series`, `summary`
  trend, CLI trend table, both dashboard pages) were updated to request
  `ascending=True`; `test_e2e_monitor.py` updated to the new contract.

Full suite green at 219 passing after the fixes.

## Honest limitations (V4)

- **OTel receiver throughput:** embedded Python gRPC server caps at ~500 spans/sec.
  Fine for solo devs and small teams; enterprise scale needs a Go collector sidecar.
- **SQLite write contention:** still the V2 scaling bottleneck; occasional
  `database is locked` under ~10 concurrent writers.
- **GenAI semconv instability:** conventions are in Development status; attribute
  names may change. The pinned version is documented.
- **Benchmark fairness:** competitor runners use default configs ŌĆö the benchmark
  measures out-of-the-box experience, not tuned performance.
- **ragas v0.4 live run unverified in this environment:** see the benchmark caveat
  above.
- **Dashboard is read-only:** a viewer, not a control plane.

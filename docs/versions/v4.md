## What's new

V4 is the first version where spanIQ becomes something other people use, not just
something built. V1-V3 were architecture; V4 is distribution.

V1 answered: "is this output good?"
V2 answered: "is my LLM still good?"

## Shipped capabilities

### OTelCollector — ecosystem entry

OTLP/gRPC (port 4317) and OTLP/HTTP (port 4318) receiver that maps OTel spans to
spanIQ `Trace` objects with `ComponentSpan` extraction. Two ingestion modes:

- **GenAI semantic conventions (auto-mapped):** `gen_ai.operation.name` maps to
  `ComponentKind` (chat / retrieval / execute_tool / invoke_agent); input/output
  from prompt/completion events; latency from span duration; error from status.
- **Generic spans (manual tags):** any span with `spaniq.input` / `spaniq.output`
  attributes is accepted, with optional `spaniq.component` / `spaniq.component_kind`.
  Lets users instrument any app, not just GenAI-convention-compliant ones.

Both modes produce the same `Trace(input, output, components=[...])` that V2/V3
already consume. The middle pipeline is untouched.

`spaniq collect-otel --baseline my_baseline` starts the receiver and pipes traces
into the Monitor.

### Streamlit local dashboard — visibility

`spaniq dashboard` launches a local Streamlit app reading directly from SQLite. No
account, no server, no cloud. Four pages:

1. **Overview** — total traces, active alerts, per-component health badge, last
   score per metric, ingestion sparkline.
2. **Drift Timeline** — interactive Plotly time-series per metric per component,
   with threshold lines. Replaces the matplotlib PNGs.
3. **Attribution** — V3's changepoint/cascade report as an interactive cascade
   visualization with root-cause and cascade markers.
4. **Alert Log** — sortable/filterable table of fired alerts, CSV export.

Read-only viewer, not a control plane. Control stays in the CLI.

### Determinism benchmark — proof

> Superseded by the accuracy benchmark (see "Accuracy benchmark rewrite" below).
> The headline claim here — "every LLM judge shows non-zero spread" — was later
> falsified by a live run and is retained only as the original record.

Reproducible benchmark suite comparing spanIQ against LLM-as-judge competitors on
variance, cost, and speed across three committed JSONL datasets (qa_factual,
summarization, rag_retrieval). The original headline: spanIQ's std dev is exactly
0.0000 across N identical runs; every LLM judge shows non-zero spread.

Runners share one interface and return the same `BenchmarkResult`. Competitors run
on the free Groq API for fairness, with a token-based production-cost estimate
reported so the table reflects real spend on a paid model.

```bash
spaniq benchmark --tool spaniq --runs 5
spaniq benchmark --tool spaniq,groq,deepeval,ragas,langfuse --runs 3
```

### Quickstart and packaging

README and docs rewritten around a 2-line "first signal" experience.
`pyproject.toml` restructured with `otel`, `dashboard`, and `benchmark` optional
dependency groups plus an `all` convenience group. Version bumped to `0.4.0`.

## Benchmark runners — completion (this cycle)

The V4 release shipped the benchmark scaffold with three working runners (spaniq,
groq, deepeval). This cycle completed the two remaining competitors so the
determinism table covers the full planned set of five tools.

### ragas runner — rewritten to the v0.4 collections API

The legacy `from ragas.metrics import faithfulness` runner was replaced with the
v0.4 collections API (`Faithfulness` from `ragas.metrics.collections`).

> **Corrected later this cycle.** This runner was originally written to score each
> sample with `single_turn_ascore(sample)` — a method the v0.4 collections API
> does not have. See "ragas runner was silently fabricating scores" below.

- Judged by Groq's LLaMA via the OpenAI-compatible endpoint through `llm_factory`,
  with a LangChain `ChatOpenAI` wrapper fallback if the v0.4 factory rejects a
  non-OpenAI client.
- Runs on `rag_retrieval.jsonl` only (faithfulness needs retrieved context); guards
  against datasets that lack context.
- Token-based production-cost estimate.
- Graceful skip (raises `ImportError` / `EnvironmentError` / `ValueError`, caught by
  the CLI) when ragas or the Groq key are missing.

### langfuse runner — Langfuse-style LLM-as-judge

Langfuse has no standalone local eval function — its LLM-as-a-Judge runs server-side
on ingested traces. This runner replicates the methodology, not the platform: the
same kind of judge prompt template a Langfuse user configures, run against Groq
locally. No Langfuse SDK or server required.

- JSON `{score, reasoning}` 1-10 template, normalized to 0-1, with loose-number
  fallback parsing.
- Concurrent batch judging via `asyncio.gather`.
- Real token-usage cost from the judge calls.
- Runs on `qa_factual.jsonl` for apples-to-apples LLM-judge variance comparison.

### Shared and supporting changes

- `benchmarks/runners/_cost.py` — shared token-based cost helper ($0.27/1M tokens).
- Both runners registered in `benchmarks/run_benchmark.py`.
- Skip + score-parser tests in `tests/test_benchmark_runners.py`.
- `pyproject.toml` ragas pin bumped to `>=0.4.0,<1.0.0`, `openai` added.
- `benchmarks/README.md` documenting both runners and the honest cost / Langfuse
  caveats.
- `benchmarks/results/` git-ignored (generated artifacts).

The honest caveat at the time: the live ragas v0.4 run could not be verified in the
build environment (no ragas installed, no Groq key). The graceful-skip path,

## Post-release fixes (found during full-suite validation)

Two pre-existing test failures were diagnosed and fixed. No API changes; both fixes
are backwards compatible.

- **ResponseDriftMetric PSI scoring** — `_discrete_psi` used `eps=1e-6` smoothing
  that made tokens missing from one distribution produce exploding `log(c/b)`
  spikes, so even an output identical to a baseline scored as drift (PSI 9.8).
  Replaced with proper Laplace (add-1) smoothing. Separately, `measure()` compared
  the output against the merged baseline corpus, inflating drift via vocabulary-size
  mismatch; it now takes the minimum PSI across individual baselines — drift is
  distance from the nearest known-good response. Identical output now scores 0.0
  (pass), clearly different output 0.43 (fail).

- **TimelineStore query ordering** — `query()` ran `ORDER BY timestamp DESC` then
  reversed the result, contradicting its own SQL, docstring, and the test. It now
  honors the DESC (newest-first) contract by default, with an explicit
  `ascending=True` flag for chronological consumers. All call sites that plot or
  compute trends over time (visualizations, sparkline, `query_series`, `summary`
  trend, CLI trend table, both dashboard pages) were updated to request
  `ascending=True`; `test_e2e_monitor.py` updated to the new contract.

Full suite green at 219 passing after the fixes.

## Accuracy benchmark rewrite (this cycle)

The determinism benchmark was rewritten into an **accuracy** benchmark, because a
live run showed the determinism headline was false: the Groq judge scored std dev
0.0000 on all three datasets at `temperature=0`, so "LLM judges vary every run"
did not hold. The new benchmark answers the question that matters — *does the tool
catch bad outputs?* — with precision / recall / F1 / AUC on labeled data.

- **Labeled datasets.** `benchmarks/datasets/build_labeled.py` derives good/bad
  rows from immutable `*_seed.jsonl` sources with a documented failure taxonomy
  (`wrong_entity`, `contradiction`, `omission`, `hallucination`,
  `unfaithful_to_context`); provenance in `datasets/LABELING.md`. qa_factual is
  100 rows balanced 50/50; summarization and rag_retrieval are 16 each.
- **Prediction contract.** Every runner now exposes `run_*_predictions` returning
  a `LabeledResult`; the good/bad label comes from the dataset, not the tool.
  Accuracy metrics live in `benchmarks/analysis/metrics.py` (self-contained
  precision/recall/F1/ROC-AUC/average-precision).
- **Fairness rule.** `benchmarks/analysis/calibrate.py` picks each tool's
  threshold by F1 on a fixed-seed calibration fold and reports on the held-out
  test fold, plus threshold-free AUC.
- **Reporting.** `report_accuracy.py` writes `accuracy.csv`, a `predictions.csv`
  audit trail (rebuild any confusion matrix by hand), and `accuracy_summary.md`.
  CLI: `spaniq benchmark --metric accuracy` (default; `--metric variance` keeps
  the legacy path).
- **Live five-tool result (2026-07-10, tracked in `benchmarks/results_accuracy/`):**
  on qa_factual the LLM judges win on accuracy (groq F1 1.0, deepeval F1 0.91)
  while spaniq's embedding metric scores F1 0.88 — the honest outcome: a
  statistical metric loses on lexically-close wrong answers. On the smaller
  summarization/rag sets the judges defaulted to passing everything (F1 0.667)
  while spaniq scored 1.0. Complementary, not interchangeable — and now measured.
- Trust document: `BENCHMARK_ACCURACY.md`; contract: `docs/plans/benchmark_v2_accuracy.md`.

## Pre-presentation blocker fixes (this cycle)

A review pass before showing the project externally turned up five issues. All are
fixed and pushed. The first is the one that matters.

### 1. The ragas runner was silently fabricating scores

Upgrading ragas to the pinned `>=0.4.0` did not make it work. Two distinct faults
were stacked on top of each other:

- **Import fault.** `ragas.llms.base` unconditionally imports
  `langchain_community.chat_models.vertexai`, which `langchain-community>=0.4`
  removed. Any ragas version fails to import against a modern langchain-community.
  Fixed by pinning `langchain-community<0.4` in the `benchmark` extra.
- **API fault (the serious one).** The runner scored with
  `single_turn_ascore(sample)` — **a method the v0.4 collections API does not
  have**. Every call raised `AttributeError`; a bare `except Exception` converted
  each one into the neutral `0.5` fallback. The runner had never actually worked.
  Had it not been skipping on the import error, the benchmark would have published
  a full column of 16 fabricated `0.5`s as if they were real judgments.

The real API is `ascore(user_input, response, retrieved_contexts) -> MetricResult`.
With it, ragas returns genuine values. Both the prediction and legacy paths were
corrected, the dead `_to_samples` helper removed, and the misleading
"collections API not available" error message replaced with the true cause.

**The structural fix:** `run_ragas_predictions` now raises `RuntimeError` when
*every* item fails, rather than reporting a full column of fallbacks. A neutral
fallback is correct for a transient timeout and dangerous for a broken
integration. Two regression tests lock this in: one asserts `Faithfulness` exposes
`ascore` and *not* `single_turn_ascore`; one proves the all-fail guard fires.

This is now limitation #0 in `BENCHMARK_ACCURACY.md`: *treat any tool whose scores
are uniformly `0.5` as broken, not as evidence.*

### 2. The e2e test enshrined the falsified headline

`tests/test_e2e_benchmark.py` asserted `variance == 0` under a docstring reading
"Core social-media claim" — the marketing line living inside the test suite, which
is exactly where a reviewer looks to see what the project actually believes. It now
exercises the accuracy path, and independently recomputes precision / recall / F1
from the raw predictions on the same held-out fold to prove the audit trail
reconciles with the reported numbers. Determinism and $0 cost remain asserted, as
the secondary by-construction facts they are.

### 3. CLI self-described as a determinism tool

`--help` printed "spanIQ determinism benchmark suite" while `--metric` already
defaulted to accuracy. Both the module docstring and the argparse description now
lead with accuracy, and every flag (`--tool`, `--dataset`, `--runs`, `--setup`,
`--output`) has help text; previously only `--metric` did.

### 4. Generated SQLite artifacts were committed

Four `.db` files sat in the repo root. Three (`spaniq_demo_*.db`) are the *default
output paths* of the demo commands — generated at runtime, committed by accident.
`test_diag.db` was unreferenced test debris. All four untracked and deleted, `*.db`
gitignored; verified a demo regenerates its own database from scratch.

### 5. Branch was never pushed

All benchmark work existed only locally. Verified `.env` was never tracked and no
`gsk_` key appears anywhere in git history, then pushed `v4-otel-dashboard`.

## Honest limitations (V4)

- **OTel receiver throughput:** embedded Python gRPC server caps at ~500 spans/sec.
  Fine for solo devs and small teams; enterprise scale needs a Go collector sidecar.
- **SQLite write contention:** still the V2 scaling bottleneck; occasional
  `database is locked` under ~10 concurrent writers.
- **GenAI semconv instability:** conventions are in Development status; attribute
  names may change. The pinned version is documented.
- **Benchmark fairness:** competitor runners use default configs — the benchmark
  measures out-of-the-box experience, not tuned performance.
- **Neutral fallbacks can fake data:** every LLM-judge runner degrades a failed
  item to `0.5`. Fine for a transient timeout, dangerous for a broken integration
  (see the ragas correction). Only the ragas prediction path currently raises when
  all items fail; the other runners would still report a column of `0.5`s.
- **ragas row not yet filled:** the integration is fixed and verified, but the
  verification run exhausted the Groq free-tier daily token budget.
- **Small n:** summarization and rag_retrieval are 16 rows each — directional only.
- **Dashboard is read-only:** a viewer, not a control plane.

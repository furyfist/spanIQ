## What's new

V3 answers the question V2 couldn't: when a multi-component LLM pipeline degrades, which component broke first, when exactly did it start, and which downstream failures are cascade?

V1 answered: "is this output good?"
V2 answered: "is my LLM still good?"
V3 answers: "WHAT broke, WHEN, and what's cascade vs root cause?"

Cost of a diagnosis: **$0.00**. Deterministic.

## New capabilities

### Per-component monitoring (`PipelineMonitor`)

Traces now carry typed component spans aligned with OpenTelemetry GenAI semantic conventions (`retrieval`, `execute_tool`, `chat`, `invoke_agent`). Each component gets its own rolling window, baseline, and V1 metric scores written to `TimelineStore`.

```python
from spaniq.attribution.component import ComponentSpan, ComponentKind
from spaniq.attribution.pipeline_monitor import PipelineMonitor
from spaniq.monitor.collectors.file import FileCollector

pm = PipelineMonitor(
    pipeline_name="rag-prod",
    collector=FileCollector("traces.jsonl"),
    window_size=20,
    warmup=20,
)
report = pm.run()
```

JSONL trace format with components:
```json
{
  "input": "What is the capital of France?",
  "output": "Paris",
  "components": [
    {"name": "retrieval", "kind": "retrieval", "output": "Paris is the capital...", "latency_ms": 120},
    {"name": "generation", "kind": "chat", "output": "Paris", "latency_ms": 900}
  ]
}
```

Backward compatible: traces without `components` behave exactly as V2 (treated as a single `default` component).

### Changepoint detection

Two algorithms, two roles:

**CUSUM (online)** — hand-implemented two-sided CUSUM (Page 1954). Updated per trace at configurable stride to handle rolling-window autocorrelation. Fires a fast alarm when a component's score series shifts. O(1) per observation.

**PELT (offline)** — exact segmentation via `ruptures` with RBF cost model (Killick et al. 2012). Runs on demand over stored timeline. Returns precise break indices. The attribution report always quotes PELT localization; CUSUM alarm index is detection latency metadata.

`h` is calibrated empirically to ARL0 >= 500 traces between false alarms.

### Failure attribution (`Attributor`)

```python
from spaniq.attribution.attributor import attribute

result = attribute(
    timeline=store,
    components=["retrieval", "generation", "search_tool"],
    metrics=["ResponseDriftMetric", "SemanticSimilarityMetric", "OutputStabilityMetric"],
    last_n=500,
)
# result.root_cause, result.cascade, result.healthy, result.verdict
```

Ranking logic: PELT changepoints per `(component, metric)` series are clustered within a proximity window (default: 10 traces). The earliest component in a cluster is the root-cause candidate. Confidence = 0.6 × (broken metrics / total) + 0.4 × (lead gap / window).

Verdict language says "broke first" — never "caused". Temporal precedence is not causal inference.

## Validation results

Measured on 200 synthetic two-component cascade scenarios with known ground-truth break points.

| shift magnitude | detection delay (median) | localization error (median) | false alarms /1k |
|---|---|---|---|
| 0.5 sigma | 79 traces | ±20 | 0.06 |
| 1 sigma | 45 traces | ±5 | 0.06 |
| 2 sigma | 17 traces | ±0 | 0.06 |
| 4 sigma | 9 traces | ±0 | 0.06 |

| lead gap | attribution accuracy |
|---|---|
| 2–4 traces | 84–90% |
| 5+ traces | 90–97% |

Attribution accuracy (lead gap >= 5 traces): **95%**
Attribution accuracy (lead gap < 5 traces): **86%** — documented limitation; small lead gaps are hard to distinguish from noise.

## Scope distinction (important)

V3 solves **population-level attribution** — systemic degradation where one component's output distribution shifts and failures cascade downstream across many traces.

V3 does **not** solve single-trace attribution ("this one run failed, which step caused it?"). That problem has no distributional signal and requires an LLM or human. Both problems are real; V3 covers the complementary, statistically tractable class.

## Post-release fixes (found during cascade demo validation)

Three bugs were found and fixed immediately after release during live validation of the cascade attribution demo. No API changes; all fixes are backwards compatible.

- `e6534ae` — **PELT penalty hardcoded to 3.0:** produced spurious changepoints almost immediately after warmup on ~180-point series. Fixed: penalty now defaults to BIC formula (`3 * log(n)`) computed from actual series length, giving ~15.3 for n=180. See `docs/decisions/pelt-penalty-calibration.md`.
- `ba3589a` — **Shared metric instance `self._window` caused cross-component contamination:** `ResponseDriftMetric` and `OutputStabilityMetric` each held an internal `deque` that accumulated outputs from all components interleaved, because `PipelineMonitor` reuses one metric instance per metric type across all components. `search_tool` (constant output, healthy control) was incorrectly scored as drifted. Fixed: removed `self._window` from both metrics; they are now stateless, scoring `actual_output` against the `baseline_outputs` passed in via `LLMTestCase`. See `docs/decisions/stateless-metrics-violation.md`.
- `a0b9b5b` + `d6c84c0` — **Attribution ranking had no clustering and no warmup offset:** PELT indices were reported in scored-series space (0-based from end of warmup) rather than real trace space, and the ranking logic declared the global minimum PELT index root cause with no clustering — a spurious early break anywhere unconditionally won. Cascade members also got `lead_gap=0` hardcoded, zeroing 40% of their confidence score. Fixed: warmup offset added to all indices; chain clustering groups nearby breaks before ranking; tie-break uses metric breadth then CUSUM alarm; cascade lead_gap computed relative to previous member in chain. See `docs/decisions/cluster-and-tiebreak-ranking.md`.
